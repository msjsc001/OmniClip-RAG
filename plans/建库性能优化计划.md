# 建库性能优化计划

## 目标

本计划的目标不是盲目把 CPU / GPU / 内存拉满，而是在保证稳定、可回退、可观察的前提下，提高大库全量建库尤其是向量阶段的吞吐。

本轮执行的硬约束：

- 未经明确要求，不构建 EXE。
- UI 只负责配置与展示，性能策略必须留在后端。
- 建库速度优化不能以牺牲稳定性为代价，任何 OOM / 压力事件都必须自动降级。
- CPU、GPU、内存必须协调考虑，不能只盯一个利用率指标。

---

## 当前判断

对 1.2 万页以上、近 30 万片段的 Logseq 笔记库来说，首轮全量建库很慢是正常现象，但“越到后边越磨、CPU/GPU 利用率都不高”的表现说明当前瓶颈并不只是模型本身，而是：

- 向量编码批次过于保守
- 编码批次与写库批次被绑死
- GPU 编码与 LanceDB 写入之间缺少更好的吞吐控制
- 后段写库追加越来越重，整体吞吐下降

所以本轮目标不是“几小时变十几分钟”，而是在不破坏稳定性的前提下，把这条链路做得更像一个受控的建库管线。

## 当前计划的完成边界

前六个阶段现在都已经完成，并且都属于当前主链的一部分。

这意味着当前版本已经具备：

- 可用的全量建库；
- 可观察的任务详情；
- 可配置的硬件峰值档位；
- 自适应批次与 OOM 回退；
- 比之前可信得多的分阶段 ETA；
- 稳定单写线程的向量写库流水线；
- “编码 / 准备行数据 / 写库 / flush”拆开的后段可观测性；
- 建库历史里的向量准备耗时、写库耗时与 flush 统计；

所以当前这份计划已经收尾完成。

如果后面还要继续投入，应被理解为：

**“在当前计划之外继续追求超大库全量建库上限的新专题”，而不是“当前软件还缺关键建库能力”。**

### 对后续投入的判断

如果你的实际使用模式更偏向：

- 首轮建好一次，
- 后续主要依赖热监听和查询，
- 很少主动重新全量建库，

那么现在继续做更深的 LanceDB 专项改进，投入产出比就会明显下降。

如果你的实际使用模式更偏向：

- 非常大的笔记库，
- 并且会**频繁执行全量建库**，
- 而且新版本实测后仍然是后段写库明显主导整体时长，

那再继续投入就仍然有意义，但那已经是下一轮独立性能专题，不是当前计划的未完成项。

## 第一阶段：建库资源控制器与峰值档位


### 目标

- 让用户用一个可理解的设置决定建库时的“激进程度”。
- 把性能控制权从 GUI 零散逻辑收回到后端。

### 已完成

- 新增 `omniclip_rag/build_control.py`。
- 新增建库硬件峰值档位：
  - `30%` 安静
  - `50%` 平衡
  - `90%` 峰值
- 新增配置字段 `build_resource_profile`，持久化到 `AppConfig`。
- GUI 已接入该设置项与 tooltip，用户可直接配置而不需要读说明书。

### 本阶段刻意不做

- 不把档位解释成精确硬件百分比锁定器。
- 不暴露一堆底层 batch / 队列参数让用户自己调。

### 是否需要重建

- 不需要。

## 第二阶段：编码 / 写库批次解耦与自适应调节

### 目标

- 不再让向量编码批次和 LanceDB 写库批次绑死。
- 在 CPU / GPU / 内存有余量时扩大吞吐，在压力升高或 OOM 时自动收缩。

### 已完成

- `vector_index.py::rebuild()` 现在把：
  - 编码批次
  - 写库批次
  视为两个独立控制量。
- 后端会基于：
  - CPU 占用
  - 内存占用
  - GPU 占用
  - GPU 显存占用
  - 最近编码耗时
  - 最近写库耗时
  自动调节当前批次。
- 已加入 OOM 安全回退：
  - 自动缩小编码批次
  - 自动缩小写库批次
  - 进入冷却窗口，避免反复抖动

### 本阶段刻意不做

- 不做激进的实时高频调节，避免 batch 大小来回震荡。
- 不做多线程 / 多进程的重型流水线改造，先以低风险方式改善吞吐。

### 是否需要重建

- 不需要。设置对下一次全量建库直接生效。

## 第三阶段：资源可观测性与任务详情展示

### 目标

- 让“为什么慢、现在在怎么调”可见，而不是只看到一个进度条。

### 已完成

- 建库阶段会在任务详情里显示：
  - 当前峰值档位
  - 当前编码批次
  - 当前写入批次
  - 资源采样摘要
  - 当前调节动作
  - 当前调节原因
- 这部分显示完全由后端快照驱动，GUI 不自行推理。

### 本阶段刻意不做

- 不把任务面板做成专业 profiler。
- 不暴露过多低层指标，避免普通用户阅读成本过高。

### 是否需要重建

- 不需要。

## 第四阶段：分阶段 ETA 动态估算

### 目标

- 让剩余时间真正利用当前阶段已经跑出来的真实速度。
- 避免 ETA 长时间停留在粗线性平均值，尤其避免在向量后半段明显失真。

### 已完成

- 新增 `BuildEtaTracker`，对 `indexing / rendering / vectorizing` 分阶段维护最近窗口吞吐。
- ETA 现在不只使用静态预估和整阶段平均值，还会使用：
  - 当前阶段最近窗口速度
  - 向量阶段尾段放缓偏置
  - 过往同配置建库历史
- 向量阶段最近尾段速度会写入 `build_history.json`，下一次相同配置建库时，初始 ETA 会更接近真实后半段成本。

### 本阶段刻意不做

- 不把 ETA 伪装成绝对精确秒表。
- 不把任务面板改成高频抖动的实时预测器，仍然保留平滑和保守边界。

### 是否需要重建

- 不需要。

---

## 第五阶段：稳定单写线程流水线并行化

### 目标

- 在不引入激进多写线程或多进程的前提下，把“读取 / 组批 / GPU 编码 / LanceDB 写入”改进成更稳定的受控流水线。

### 已完成

- `vector_index.py::rebuild()` 现在采用“主线程编码 + 单写线程落库 + 有界队列缓冲”的稳定流水线。
- 编码阶段不再每批都同步等待写库完成，写线程会在后台分批 flush，减轻后段串行等待。
- 写入队列容量是受控的，不会无限膨胀占满内存。
- 暂停 / 取消 / OOM 回退 / 最终 drain 仍然保持单路径收口，避免把稳定性换成表面吞吐。

### 本阶段刻意不做

- 不做多写线程竞争同一向量表。
- 不做多进程 fan-out 编码。
- 不做为了吃满硬件而牺牲暂停、取消、恢复一致性的激进并行。

### 为什么这样做

- 真正的大库问题是后段越来越慢，而不是前段完全没有速度。
- 对当前产品而言，最有价值的是先把“编码与写入串行等待”改进成稳定流水线，而不是直接上复杂并行架构。
- 这一阶段已经能明显改进大库全量建库后半段的空转和等待，但不会破坏现有正确性边界。

### 是否需要重建

- 不需要。对下一次全量建库直接生效。

---

## 第六阶段：更细粒度的后段写库优化

### 目标

- 继续改进大库后半段 LanceDB 追加越来越慢的问题。

### 已完成

- 已把向量尾段最近速度写入 `build_history.json`，用于下一次建库和 ETA 初始估算。
- 已把 ETA 从单纯的全局平均值提升为“静态基线 + 当前阶段最近窗口 + 向量尾段偏置”的组合估算。
- 已把写入队列深度、已编码数量、已写入数量、flush 次数纳入向量阶段进度与任务详情，便于判断后段瓶颈是否正在转向写库。
- 已把写入队列积压纳入资源控制器决策，允许在保持稳定的前提下优先改进写库批次，而不是一味扩大编码批次。
- 已把“准备行数据”和“真正写库”拆开统计，并把这两段都写入建库历史。
- 已把向量阶段改成“主线程编码 + 单写线程准备/写库 + 有界队列”的稳定流水线，减轻大库后段串行等待。

### 本阶段刻意不做

- 不做多写线程争用同一向量表。
- 不做更激进的多进程编码与写库并发。
- 不为了追求极限峰值而牺牲暂停、取消、OOM 回退与一致性边界。

### 为什么这样收口

- 这一阶段的目标是稳定改进大库后段越来越慢的问题，而不是把系统改造成高风险性能实验场。
- 当前已经把后段问题拆到足够可观察、可调节、可回退的程度，适合作为长期主链能力保留。

## 预期收益评估

### 这一轮已完成项的意义

这轮不是“极限加速”，而是先把建库过程从“固定保守小批量”升级为“受控的自适应大批量”。

### 预估提升

在你这种：

- 1.2 万页以上
- 近 30 万片段
- GPU 可用
- 越到后边越慢

的场景下，这轮更现实的预期是：

- 建库总时长有机会提升 `10% ~ 35%`
- 更重要的是：
  - GPU 吃得更稳
  - 后段不会那么保守
  - OOM 风险更低
  - 用户不用自己反复试 batch 大小

### 为什么不是承诺 2x / 3x

因为当前瓶颈里，除了编码批次保守，还有：

- Python 层对象转换
- Arrow 序列化
- LanceDB 后段追加
- Windows 本地 I/O

这些都还没有被彻底重构。所以这轮会明显更合理，但不会魔法级提速。

---

## 这轮后的判断

本轮已经把“建库性能控制器”打进了主链，主要解决的是：

- 用户可配置建库激进程度
- 后端自动按 CPU / GPU / 内存协同调节
- OOM 和压力事件自动回退
- UI 与性能逻辑保持解耦

后续如果要继续往上追求更大幅度的提速，优先级应该是：

1. 真实大库 profiling
2. 深度流水线并行化
3. 后段 LanceDB 写入专项优化
4. 再评估是否需要更激进的 GPU 管线
