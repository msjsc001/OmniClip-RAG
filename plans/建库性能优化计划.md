# 建库性能优化计划

## 目标

本计划的目标不是盲目把 CPU / GPU / 内存拉满，而是在保证稳定、可回退、可观察的前提下，提高大库全量建库尤其是向量阶段的吞吐。

本轮执行的硬约束：

- 未经明确要求，不构建 EXE。
- UI 只负责配置与展示，性能策略必须留在后端。
- 建库速度优化不能以牺牲稳定性为代价，任何 OOM / 压力事件都必须自动降级。
- CPU、GPU、内存必须协调考虑，不能只盯一个利用率指标。

---

## 当前判断

对 1.2 万页以上、近 30 万片段的 Logseq 笔记库来说，首轮全量建库很慢是正常现象，但“越到后边越磨、CPU/GPU 利用率都不高”的表现说明当前瓶颈并不只是模型本身，而是：

- 向量编码批次过于保守
- 编码批次与写库批次被绑死
- GPU 编码与 LanceDB 写入之间缺少更好的吞吐控制
- 后段写库追加越来越重，整体吞吐下降

所以本轮目标不是“几小时变十几分钟”，而是在不破坏稳定性的前提下，把这条链路做得更像一个受控的建库管线。

## 第一阶段：建库资源控制器与峰值档位

### 目标

- 让用户用一个可理解的设置决定建库时的“激进程度”。
- 把性能控制权从 GUI 零散逻辑收回到后端。

### 已完成

- 新增 `omniclip_rag/build_control.py`。
- 新增建库硬件峰值档位：
  - `30%` 安静
  - `50%` 平衡
  - `90%` 峰值
- 新增配置字段 `build_resource_profile`，持久化到 `AppConfig`。
- GUI 已接入该设置项与 tooltip，用户可直接配置而不需要读说明书。

### 本阶段刻意不做

- 不把档位解释成精确硬件百分比锁定器。
- 不暴露一堆底层 batch / 队列参数让用户自己调。

### 是否需要重建

- 不需要。

---

## 第二阶段：编码 / 写库批次解耦与自适应调节

### 目标

- 不再让向量编码批次和 LanceDB 写库批次绑死。
- 在 CPU / GPU / 内存有余量时扩大吞吐，在压力升高或 OOM 时自动收缩。

### 已完成

- `vector_index.py::rebuild()` 现在把：
  - 编码批次
  - 写库批次
  视为两个独立控制量。
- 后端会基于：
  - CPU 占用
  - 内存占用
  - GPU 占用
  - GPU 显存占用
  - 最近编码耗时
  - 最近写库耗时
  自动调节当前批次。
- 已加入 OOM 安全回退：
  - 自动缩小编码批次
  - 自动缩小写库批次
  - 进入冷却窗口，避免反复抖动

### 本阶段刻意不做

- 不做激进的实时高频调节，避免 batch 大小来回震荡。
- 不做多线程 / 多进程的重型流水线改造，先以低风险方式改善吞吐。

### 是否需要重建

- 不需要。设置对下一次全量建库直接生效。

---

## 第三阶段：资源可观测性与任务详情展示

### 目标

- 让“为什么慢、现在在怎么调”可见，而不是只看到一个进度条。

### 已完成

- 建库阶段会在任务详情里显示：
  - 当前峰值档位
  - 当前编码批次
  - 当前写入批次
  - 资源采样摘要
  - 当前调节动作
  - 当前调节原因
- 这部分显示完全由后端快照驱动，GUI 不自行推理。

### 本阶段刻意不做

- 不把任务面板做成专业 profiler。
- 不暴露过多低层指标，避免普通用户阅读成本过高。

### 是否需要重建

- 不需要。

---

## 第四阶段：分阶段 ETA 动态估算

### 目标

- 让剩余时间真正利用当前阶段已经跑出来的真实速度。
- 避免 ETA 长时间停留在粗线性平均值，尤其避免在向量后半段明显失真。

### 已完成

- 新增 `BuildEtaTracker`，对 `indexing / rendering / vectorizing` 分阶段维护最近窗口吞吐。
- ETA 现在不只使用静态预估和整阶段平均值，还会使用：
  - 当前阶段最近窗口速度
  - 向量阶段尾段放缓偏置
  - 过往同配置建库历史
- 向量阶段最近尾段速度会写入 `build_history.json`，下一次相同配置建库时，初始 ETA 会更接近真实后半段成本。

### 本阶段刻意不做

- 不把 ETA 伪装成绝对精确秒表。
- 不把任务面板改成高频抖动的实时预测器，仍然保留平滑和保守边界。

### 是否需要重建

- 不需要。

---

## 第五阶段：深度流水线并行化

### 目标

- 进一步把“读取 / 组批 / GPU 编码 / LanceDB 写入”解耦成更完整的流水线。

### 当前未做

- 还没有上真正的 producer / consumer 多阶段并行。
- 目前仍然是保守的单进程受控管线，只是把写库缓冲和批次控制做得更合理。

### 为什么先不做

- 这一层复杂度明显更高。
- 如果先上重型流水线，调试成本、数据一致性风险、暂停/取消/恢复的复杂度都会陡增。

---

## 第六阶段：更细粒度的后段写库优化

### 目标

- 继续压缩大库后半段 LanceDB 追加越来越慢的问题。

### 已完成

- 已把向量尾段最近速度写入 `build_history.json`，用于下一次建库和 ETA 初始估算。
- 已把 ETA 从单纯的全局平均值提升为“静态基线 + 当前阶段最近窗口 + 向量尾段偏置”的组合估算。

### 当前未做

- 还没有针对 LanceDB 后段大表追加做更细粒度 profiling 和专门优化。
- 还没有按阶段拆出“编码耗时 vs Arrow 序列化 vs LanceDB 写入耗时”的独立统计面板。

### 为什么先不做

- 需要先在真实大库上观察这轮控制器的效果，再决定瓶颈是不是已经转移。

---

## 预期收益评估

### 这一轮已完成项的意义

这轮不是“极限加速”，而是先把建库过程从“固定保守小批量”升级为“受控的自适应大批量”。

### 预估提升

在你这种：

- 1.2 万页以上
- 近 30 万片段
- GPU 可用
- 越到后边越慢

的场景下，这轮更现实的预期是：

- 建库总时长有机会提升 `10% ~ 35%`
- 更重要的是：
  - GPU 吃得更稳
  - 后段不会那么保守
  - OOM 风险更低
  - 用户不用自己反复试 batch 大小

### 为什么不是承诺 2x / 3x

因为当前瓶颈里，除了编码批次保守，还有：

- Python 层对象转换
- Arrow 序列化
- LanceDB 后段追加
- Windows 本地 I/O

这些都还没有被彻底重构。所以这轮会明显更合理，但不会魔法级提速。

---

## 这轮后的判断

本轮已经把“建库性能控制器”打进了主链，主要解决的是：

- 用户可配置建库激进程度
- 后端自动按 CPU / GPU / 内存协同调节
- OOM 和压力事件自动回退
- UI 与性能逻辑保持解耦

后续如果要继续往上追求更大幅度的提速，优先级应该是：

1. 真实大库 profiling
2. 深度流水线并行化
3. 后段 LanceDB 写入专项优化
4. 再评估是否需要更激进的 GPU 管线
