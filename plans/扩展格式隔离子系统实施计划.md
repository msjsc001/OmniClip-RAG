# 扩展格式隔离子系统实施计划

## 文档定位

这份文档是“扩展格式”功能的唯一执行蓝图与持续更新底稿。

后续无论在任何聊天窗、任何开发阶段继续推进扩展格式能力，都必须先阅读本文件，再继续实施。

本文件的目的不是写概念，而是确保以下事情始终成立：

- 新功能在架构上高度解耦；
- 新功能不会污染当前 Markdown 主链；
- 中途换聊天窗、换上下文、甚至上一次对话完全卡死时，也能从这份文档直接接着做；
- 每一阶段做完后，都能把真实进展、真实问题、真实妥协继续写回这里；
- 最终上线时，可用这份文档对照验收，不会“做着做着变形”。

## 当前背景快照

### 当前产品主链

当前软件的稳定主链是：

- 以 Markdown / Logseq 笔记库为核心输入；
- 通过解析、切块、SQLite + FTS5、LanceDB 向量索引完成本地混合检索；
- Qt 为主 UI；
- 打包版为轻量 onedir；
- 运行时数据写入 `%APPDATA%\OmniClip RAG`；
- 当前已重点打磨：
  - 全量建库；
  - 热监听；
  - 中断恢复；
  - OOM / 高压回退；
  - 构建与运行时分离；
  - 版本化打包目录；
  - 双语 UI；
  - 日志与诊断。

### 当前要新增的能力

在现有 Markdown 主链之外，新增一个完全独立的“扩展格式”子系统，第一批收敛为两条能力线：

- PDF（专门解析链，优先 `pypdf`）
- Tika 管理的多格式扩展链（外置 sidecar 运行时）

### 当前已经明确的产品方向

- 扩展格式不是往现有 Markdown 主系统里塞几个 parser。
- 扩展格式要以“隔离子系统”的方式出现。
- 隔离范围包括：
  - UI
  - 配置
  - 数据目录
  - 建库状态
  - 热监听状态
  - 删除/重建
  - 日志标识
- 但会复用现有成熟基建：
  - 任务框架
  - 进度上报
  - 暂停/取消
  - 日志
  - watchdog
  - 预检框架
  - 查询结果融合框架
  - 数据目录总规则
  - UI 样式体系

## 绝对硬约束

以下是不可妥协约束。任何实现只要违反其中任意一条，都视为方案失败。

### 1. 不能破坏当前已有功能

扩展格式功能上线后，即使用户完全不使用它，当前软件的以下行为也必须与现在保持一致：

- Markdown 主建库流程；
- Markdown 热监听；
- 现有查询行为；
- 当前数据目录与索引状态判断；
- 当前打包体积控制策略；
- 当前运行时引导逻辑；
- 当前日志、清理、配置加载；
- 当前 UI 主流程。

执行标准：

- 扩展格式默认关闭；
- 关闭时主链行为必须等价于“此功能不存在”；
- 如出现任何主链行为回归，该阶段不得合入。

### 2. 不能污染主索引状态

扩展格式任何一种格式建库失败、监听失败、配置错误、来源目录失效，都不能影响：

- Markdown 主索引是否存在；
- Markdown 主库是否可查询；
- Markdown 主监听是否正常；
- Markdown 主任务状态显示；
- Markdown 主状态文件。

### 3. 不能删除用户原始文件

无论是删除索引、禁用某能力线、移除来源目录，系统都只能操作自身数据目录，不得删除任何用户原始扩展文档文件。

### 4. 打包体积必须受控

扩展格式的第一版设计必须遵守：

- 不能引入 OCR；
- 不能引入浏览器内核；
- 不能引入重量级网页渲染运行时；
- 不能显著膨胀打包版体积；
- 目标仍保持在轻量分发范围内，理想上不超过 `200 MB`。

### 5. 所有扩展格式都必须是“可撤销”的

每种格式都必须支持：

- 启用 / 禁用；
- 一次扫描变动更新；
- 建立索引；
- 重建索引；
- 删除该格式索引；
- 停止该格式热监听；
- 清除该格式状态；
- 不影响别的格式和 Markdown 主链。

### 6. 目录取消勾选不允许静默删数据

如果用户在某个扩展格式页签里取消勾选某个来源目录，这个动作语义上等同于“清除该目录对应的扩展索引数据”。

该动作必须满足：

- 必须弹出二次确认窗口；
- 必须明确告诉用户“仅清除该目录对应的扩展格式索引数据，不删除原始文件”；
- 必须支持用户取消；
- 未确认前不得落盘、不得删索引、不得改状态。

### 7. 源路径异常时不允许自动清库

如果来源目录因为以下原因暂时不可访问：

- 硬盘掉盘；
- 移动硬盘未接入；
- 加密盘锁盘；
- 网络盘临时断开；
- 权限暂时失效；

系统只能把该来源标记为“源路径异常 / 暂时失联”，不得自动执行以下任何动作：

- 自动删除该来源索引；
- 自动把该目录视为用户主动移除；
- 自动把该格式整体标记为未建库；
- 自动清空可恢复状态。

### 8. 扩展子系统必须通过统一协调机制与主链协作

扩展格式虽然必须与主链隔离，但不能各跑各的。

必须存在统一任务协调器，负责协调：

- Markdown 主建库；
- Markdown 主热监听重任务；
- 扩展格式全量建库；
- 扩展格式一次扫描变动更新；
- 扩展格式重建；
- 扩展格式热监听回压与恢复。

该协调器的目标不是耦合功能，而是确保扩展功能永远不能拖垮主程序。

## 核心设计原则

### 原则 A：物理隔离，逻辑融合

扩展格式底层要“物理隔离”，查询体验可以“逻辑融合”。

含义：

- 各格式各自建库；
- 各格式各自监听；
- 各格式各自状态；
- 查询时再统一聚合结果。

### 原则 B：先做隔离骨架，再接格式解析

禁止一上来就同时把 PDF/HTML/MHTML 全部塞入主链。

正确顺序：

1. 先搭扩展格式子系统骨架；
2. 再接入第一批格式；
3. 再接入独立监听；
4. 最后做查询融合收尾。

### 原则 C：现有成熟能力优先复用

凡是当前主链已经成熟的机制，不重复造轮子：

- Worker
- 任务状态
- 预检
- 暂停/取消
- 日志
- watchdog
- 清理入口
- 打包流程

但所有“状态存储与任务身份”必须做命名空间隔离。

### 原则 D：结果展示必须带来源身份

所有扩展格式结果都必须明确标识来源类型和具体来源名。

例如：

- `Markdown · 项目笔记`
- `PDF · Deep Learning.pdf · 第 12 页`
- `HTML · docs.python.org · asyncio`
- `MHTML · OpenAI 文章归档`

特别要求：

- PDF 结果不能只显示 `PDF`，必须至少显示 PDF 文件名；
- 如果能够可靠拿到页码，则必须显示页码；
- 查询融合后也必须保留该来源身份。

### 原则 E：主链与扩展链必须有统一协调器

扩展格式不是“完全无关的第二套程序”，而是主程序中的平行子系统。

因此必须同时满足：

- 状态与数据物理隔离；
- 任务调度与资源争抢统一协调；
- UI 上对用户呈现为可理解的一致行为；
- 任何扩展任务都不能越过协调器直接和主链抢占重资源。

### 原则 F：优先使用成熟轻量工具，不自造底层轮子

扩展格式功能优先复用社区成熟、轻量、被长期验证过的解析工具链。

除非没有可靠替代方案，否则不自行实现：

- PDF 文本抽取器；
- HTML DOM 解析器；
- MHTML / MIME 拆包底层；
- 文档格式转换底层。

目标是同时满足：

- 更低 bug 面；
- 更低维护成本；
- 更小打包体积；
- 更稳定的长期演进能力。

### 原则 G：Tika-first，但 PDF 例外

扩展格式默认采用 `Tika` 作为统一的多格式扩展运行时，但 `PDF` 不与 Tika 混在一起。

也就是说：

- `PDF` 走单独专门链；
- `Tika` 负责其白名单内的多格式；
- Tika 支持差的格式必须在 UI 中可见但灰色禁用；
- Tika 兼容性未知的格式只能带风险提示暴露；
- 任何需要高保真结构恢复的高频格式，都不应因为“Tika 也能解析”就强行归入 Tika 路线。

## 第一版范围定义

### 第一版支持

- `PDF` 专门解析链（优先 `pypdf`）
- `Tika` 外置扩展运行时
- `Tika` 运行时自动安装 / 手动安装 / 版本检测
- `Tika` 格式选择弹窗（带搜索）
- `Tika` 一级推荐格式直接可选
- `Tika` 兼容性未知格式可见并带风险提示
- `Tika` 兼容性差格式灰色可见但不可启用
- 独立来源目录管理
- 预检建库时间空间
- 一次扫描变动更新
- 独立建库 / 重建 / 删除
- 独立热监听
- 查询融合
- 来源过滤
- 独立日志与状态

### 第一版不做

- OCR
- 扫描件识别
- `PDF` 通过 Tika 走统一解析
- 强行暴露 Tika 兼容性差的格式
- 浏览器渲染网页
- 远程网页抓取
- 跨工作区混合源
- 与 Markdown 共用底层索引库

## UI 方案

### 位置

在“配置”中新增独立页签：

- `扩展格式`

### 顶部固定区域

顶部固定区不随子页签变化，统一保留：

1. 页头说明
2. 状态总览
3. 全局操作

### 顶部说明文案方向

建议文案方向：

> 这里用于管理 PDF 专门解析链与 Tika 多格式扩展运行时。它们独立预检、独立建库、独立删除、独立监听，但查询时可以统一融合。

### 状态总览

状态总览样式应与“开始”页的状态芯片风格一致。

主芯片至少包括：

- `PDF：未启用 / 未建库 / 已建立 / 监听中 / 出错`
- `Tika：未安装 / 已安装 / 版本过旧 / 已就绪 / 出错`
- `Tika 格式：已启用 6 项 / 已建立 4 项 / 2 项待建库`

总览区域还应额外给出汇总：

- `扩展已启用 2 条能力线 / 已建立 5 个来源索引`

### 全局操作

全局操作统一保留在顶部：

- `预检建库时间空间`
- `建立全部已启用格式索引`
- `停止当前扩展任务`

说明：

- 按用户要求，原先讨论中的 `扫描已启用格式 / 扫描文件` 统一改为 `预检建库时间空间`；
- 该语义与当前主链一致；
- 不再额外造一个“扫描文件”概念。

### 子页签结构

- `PDF`
- `Tika`

这样既能让 `PDF` 保持专门链，也能让 `Tika` 统一承载多格式扩展。

### Tika 格式选择窗

`Tika` 不能把“它能解析的所有格式”都等价暴露给用户。

因此必须提供独立格式选择弹窗，具备以下能力：

- 搜索框；
- 高频格式置顶；
- 一级推荐格式：正常可选；
- 兼容性未知格式：可见、默认关闭、启用前二次提示；
- 兼容性差格式：灰色可见但不可用；
- `PDF` 不出现在 Tika 格式选择窗中；
- 用户最终只管理“我们允许暴露的格式子集”，而不是 Tika 全量宣称支持的全部格式。

### 来源目录管理规则

来源目录必须支持多选，但要守住隔离边界。

允许两类来源：

1. 当前工作区关联目录
2. 额外自定义目录

额外规则：

- 如果某个“内置可勾选目录”已经从当前工作区移除，则扩展格式里的关联项也必须同步移除；
- 不允许直接引用其他工作区的目录配置；
- 不在第一版支持跨工作区联合来源；
- 用户主动取消勾选某个目录时，必须先弹确认窗，再清除该目录对应的扩展索引数据；
- 目录临时不可访问时，只能标记为“源路径异常”，不得等同于“用户已移除目录”；
- 只有用户明确确认移除，才允许执行目录级清理。

## 数据与状态隔离设计

### 数据目录结构

建议目录结构如下：

```text
%APPDATA%/OmniClip RAG/
└─ workspaces/
   └─ <workspace-id>/
      ├─ markdown/
      │  ├─ sqlite/
      │  ├─ state/
      │  └─ exports/
      └─ extensions/
         ├─ pdf/
         │  ├─ config.json
         │  ├─ state/
         │  ├─ sqlite/
         │  ├─ vector/
         │  ├─ logs/
         │  └─ exports/
         └─ tika/
            ├─ runtime/
            │  ├─ jre/
            │  ├─ tika-server.jar
            │  └─ runtime_state.json
            ├─ catalog/
            │  ├─ format_catalog.json
            │  └─ enabled_formats.json
            ├─ formats/
            │  ├─ docx/
            │  │  ├─ state/
            │  │  ├─ sqlite/
            │  │  ├─ vector/
            │  │  └─ logs/
            │  ├─ html/
            │  └─ ...
            └─ exports/
```

### 每种能力线独立状态文件

至少独立维护：

- `config`
- `index_state`
- `build_state`
- `watch_state`
- `stats`
- `preflight_history`
- `runtime_state`（Tika 专用）
- `enabled_formats`（Tika 专用）
- `format_catalog`（Tika 专用）

禁止行为：

- 禁止扩展格式与 Markdown 共用同一份 `index_state.json`
- 禁止扩展格式共用 Markdown 的 `watch_state.json`
- 禁止扩展格式把自身结果写入主 Markdown 索引目录
- 禁止把 `Tika` 运行时状态写回 Markdown 主链配置

### 删除语义

删除动作必须拆成三种，不得混淆：

1. `禁用该能力线或子格式`
   - 不参与查询
   - 不删除索引

2. `删除该能力线或子格式索引`
   - 只删除其在 AppData 下的索引与状态
   - 不删除原始文件

3. `移除该能力线或子格式来源配置`
   - 只移除目录配置
   - 不删除原始文件

### 目录级状态语义

每个来源目录都必须有独立状态，不能只看“这个能力线整体是否已建库”。

建议至少支持以下目录级状态：

- `enabled`
- `disabled`
- `missing_temporarily`
- `removed_confirmed`
- `stale`
- `error`

语义要求：

- `missing_temporarily` 只表示当前不可访问，不触发自动清库；
- `removed_confirmed` 只在用户明确确认取消勾选后出现；
- `stale` 表示目录内容可能发生变化，等待“一次扫描变动更新”或重建；
- 目录级状态变化不得直接污染该能力线整体 `ready / pending / error` 的定义。

### 一次扫描变动更新

每种扩展能力都必须支持“手动触发的一次扫描变动更新”。

它的定位是：

- 不是全量重建；
- 不是持续热监听；
- 而是用户在资料相对稳定时，手动触发的一次性增量同步。

行为要求：

1. 只扫描当前已启用且可访问的来源目录；
2. 检测新增、修改、删除三类变化；
3. 对变化部分做增量更新；
4. 完成后任务结束，不继续常驻监听；
5. 遇到临时失联目录时，只记录异常并跳过，不清库；
6. 最终产出目录级摘要，告诉用户哪些目录已更新、哪些目录异常、哪些目录未变化。

这项能力是第一版标准功能，不得推迟到未来版本。

- Tika 全量建库
- Tika 一次扫描变动更新
- Tika 重建

协调目标：

- 保证扩展功能零干扰主链；
- 防止多重任务同时拉高 RAM / VRAM / IO；
- 保证用户总能从 UI 看懂当前是谁在占用资源。

#### `query.py`

负责：

- 从 `Markdown / PDF / Tika` 三条能力线拉取结果；
- 对 Tika 结果补入具体子格式身份；
- 合并结果；
- 写入来源身份；
- 给主查询层返回统一结构。

#### `preflight.py`

负责：

- 对单能力线或全部已启用扩展能力做预检；
- 复用现有预检思路；
- 但计算来源目录、文件数、体积、估时时必须按能力线隔离。

#### `normalizers/*`

- `parsers/pdf.py` 只负责 PDF 专门链；
- `normalizers/pdf.py` 负责 PDF 页码、断句、去噪标准化；
- `normalizers/tika_output.py` 负责把 Tika 输出标准化为主程序统一中间结构。

任何 Tika 管理格式都不应绕过标准化层直接写入索引。

### 必须复用但不得污染的现有能力

以下能力应复用：

- Worker/任务框架
- 日志
- 任务进度快照
- 暂停 / 取消
- watchdog
- 资源控制器
- 预检展示样式
- 状态芯片样式

但扩展格式的任务 ID、状态文件、存储命名空间必须独立。

## 查询融合设计

### 目标

查询体验统一，但底层索引物理隔离。

### 设计方式

采用 broker 模式：

1. Markdown 查询服务
2. PDF 查询服务
3. Tika 查询服务

由上层统一调度，并做最终融合。

### 结果结构要求

每条结果必须带：

- `source_family`
  - `markdown`
  - `pdf`
  - `tika`
- `source_kind`
  - 例如 `docx` / `html` / `rtf` / `epub`
- `source_label`
  - 例如 `PDF · Deep Learning.pdf · 第 12 页`
  - 或 `DOCX(Tika) · 项目方案.docx`
- `source_name`
  - 原文件名或来源标题
- `source_path`
- `page_no`（如可用）
- `section_heading`（如可用）

### 查询页过滤建议

建议至少支持两层过滤：

- 一级：`全部 / Markdown / PDF / Tika`
- 二级：Tika 内部按具体格式筛选（可后置到高级筛选）

基础版本也可先用多选：

- `[√] Markdown`
- `[√] PDF`
- `[√] Tika`

### 统一排序要求

即使不同来源融合，排序也必须继续使用统一可解释的最终分数逻辑。

禁止行为：

- 仅因为结果来自 PDF 就先天降低或抬高权重；
- 仅因为来源类型不同而做不可解释排序；
- 直接把不同 LanceDB 表返回的原始向量距离分数混在一起做全局排序。

### 多表向量分数对齐原则

扩展格式采用物理隔离后，Markdown、PDF 和 Tika 子格式会各自拥有独立的向量表。不同表的数据分布不同，原始 distance / similarity 分数不能直接横向比较。

因此跨来源查询必须采用“两段式融合”：

1. 每个来源内部先独立召回并形成本来源候选；
2. 每个来源内部先完成本地排序；
3. 跨来源融合时优先依赖以下信号，而不是直接使用跨表 raw distance：
   - 来源内 rank
   - 来源内归一化分数
   - SQLite / FTS5 字面命中分数
   - 结构化命中信号
   - 可选 reranker 的统一结果

第一版建议采用稳妥策略：

- 向量 raw score 只在“单来源内部”参与排序；
- broker 融合时以 rank-based fusion 为主；
- FTS5 / lexical score 作为跨来源校验信号；
- 如果启用了 reranker，则 reranker 作为最终统一裁决层。

## 热监听设计

### 总原则

热监听必须独立。

每一种格式的监听：

- 独立开关
- 独立状态
- 独立日志
- 独立恢复
- 独立报错

### 复用要求

应尽量复用现有成熟热监听框架，包括：

- 稳定窗口
- 快照 diff
- 删除确认
- 失败回退
- watch state 持久化

### 风险控制

第一版虽然支持扩展格式热监听，但必须限制重任务并发：

- Markdown 全量建库
- Markdown 一次扫描变动更新或重型恢复任务
- PDF 全量建库
- PDF 一次扫描变动更新
- PDF 重建
- Tika 全量建库
- Tika 一次扫描变动更新
- Tika 重建

这些属于重任务，应由全局任务协调器做串行或受控调度，避免资源争抢。

扩展格式监听不得无脑在重建库过程中同时启动大规模重建。

## 包体与依赖约束

### 体积控制目标

扩展格式加入后，打包版仍应尽量维持轻量，目标不超过 `200 MB`。

### 第一版依赖选择原则

总体原则：

- 主程序继续保持轻量；
- `Tika` 以外置运行时方式存在，不打进主包；
- `PDF` 专门链优先 `pypdf`；
- 任何新增依赖都要先评估对构建包体、启动速度和长期维护成本的影响。

#### PDF

优先考虑：

- `pypdf`

原因：

- 对 `PDF` 的控制粒度更高；
- 更适合接我们自己的页码、伪断句、正文清洗与去噪链；
- 不把高频 `PDF` 质量绑死在 Tika 的统一输出上。

禁止：

- OCR 引擎
- 图像识别库
- 扫描件识别栈

#### Tika 扩展运行时

运行时约束：

- 使用外置 `tika-server.jar`；
- 使用外置 JRE；
- 运行在独立进程；
- 主程序只负责安装、检测、启动、停止和标准化输出；
- `Tika` 版本必须固定到安全版本，不允许漂移到未知版本。

格式策略：

- 一级推荐格式：直接暴露并支持；
- 兼容性未知格式：可见但带风险提示；
- 兼容性差格式：灰色可见但不可启用；
- `PDF` 不纳入 Tika 暴露白名单。

### 打包红线

- 不得把 `Tika` 运行时直接打入主包；
- 不得把额外大模型打包进去；
- 不得把文档转换重依赖拉入主包；
- 不得因为支持扩展格式而破坏当前轻量发布策略。

## 致命风险清单与防护

### 风险 1：主链被污染

#### 表现

- 扩展格式失败后主 Markdown 显示“索引损坏”；
- 主查询异常；
- 主监听状态错乱。

#### 防护

- 独立状态文件；
- 独立索引目录；
- 关闭时行为等价测试；
- 所有扩展入口默认关闭。

### 风险 2：多重任务争抢导致资源崩溃

#### 表现

- Markdown 建库与扩展重任务同时跑；
- RAM/VRAM 被同时拉爆；
- 用户误以为系统坏掉。

#### 防护

- 建立全局任务协调器；
- 重任务串行；
- 监听回压；
- 保留现有 watchdog / 降批 / 诊断。

### 风险 3：来源目录重复导致重复索引

#### 表现

- 同一文件被两个来源目录同时命中；
- 查询中重复出现；
- 索引膨胀。

#### 防护

- 规范化路径去重；
- 文件指纹去重；
- 来源映射表。

### 风险 4：单文件解析失败中断整轮任务

#### 表现

- 某个脏文件直接把全部扩展建库打断。

#### 防护

- 单文件失败只记日志并跳过；
- 整轮任务继续；
- 失败计数出现在状态摘要里。

### 风险 5：目录临时失联被误判成应清库

#### 表现

- 移动硬盘没插；
- 加密盘暂时锁住；
- 网络盘一时断开；
- 系统把目录误判成“已不存在”，并直接清掉索引。

#### 防护

- 目录级状态必须区分 `missing_temporarily` 与 `removed_confirmed`；
- 临时失联只在 UI 提示“源路径异常”；
- 只有用户明确取消勾选并确认后，才允许执行目录级清理；
- 一次扫描变动更新与热监听都必须把“临时失联”视为可恢复异常，而不是删除事件。

### 风险 6：Tika 运行时缺失、过旧或异常退出

#### 表现

- 用户启用了 Tika 格式，但未安装 JRE 或 `tika-server.jar`；
- Tika 版本低于安全要求；
- sidecar 崩溃后用户误以为主程序坏掉。

#### 防护

- `Tika` 运行时状态单独显示；
- 自动安装 / 手动安装 / 重新检测三条路径都要存在；
- 版本固定到安全版本；
- sidecar 异常只影响 Tika 路线，不影响 PDF 与 Markdown。

### 风险 7：Tika 兼容性差格式被误暴露

#### 表现

- 某些理论可解析但实际效果差的格式被正常开放；
- 用户拿它做知识库后得到大量噪声或坏结构。

#### 防护

- 必须有格式分层目录；
- 兼容性差格式只允许灰色可见；
- 兼容性未知格式启用前必须二次提示；
- `PDF` 明确不进入 Tika 白名单。

### 风险 8：跨表向量分数失真导致融合排序不公

#### 表现

- Markdown、PDF、Tika 子格式的向量距离分数被直接混排；
- 某个来源因为自身表分布更集中而长期霸榜。

#### 防护

- 不允许跨表直接比较 raw distance；
- 必须先做来源内排序，再做跨来源 rank-based fusion；
- FTS5 / lexical score 作为跨来源二次校验；
- 如启用 reranker，则以 reranker 作为最终统一裁决层。

### 风险 9：PDF 物理行抽取导致语义碎裂

#### 表现

- 一句话在页尾断开，在下一页续上；
- 直接切块后全是半句；
- 查询命中虽然有字面重合，但语义向量质量极差。

#### 防护

- PDF 必须先经过“文本规范化 + 伪断句缝合”层，再进入切块；
- 缝合逻辑至少要处理行内断句、跨行续接、必要的跨页续接；
- 仍要保留页码跨度元信息。

### 风险 10：新增格式时架构分叉、反向污染主程序

#### 表现

- 新增格式时直接复制逻辑到主链；
- 某个格式为了省事直接写入 Markdown 主状态；
- 扩展功能越做越像主程序里的大量条件分支。

#### 防护

- 新格式必须继续挂在 `extensions/` 命名空间下；
- 新格式必须遵守同一套目录级状态语义、协调器和查询 broker；
- 新格式接入前必须先验证“关闭时主链零回归”；
- 任何需要修改主链默认行为的实现都视为高风险方案，默认否决。

### 风险 11：聊天窗切换导致上下文丢失

#### 防护

- 本文档必须持续更新；
- 每完成一个阶段，必须把真实状态写回本文件；
- 新聊天窗继续时，以本文件为唯一权威背景。

## 阶段 3：PDF 专门解析链与独立建库

### 目标

支持文本型 PDF，并明确不走 Tika。

### 交付

- PDF 文本抽取；
- 页码元信息；
- PDF 文本规范化层；
- 伪断句缝合逻辑；
- 独立预检；
- 一次扫描变动更新；
- 独立建库 / 重建 / 删除；
- 查询展示为 `PDF · 文件名 · 第 N 页`

### PDF 解析硬约束

PDF 不能按物理行原样直接切块。必须在切块前做一次轻量启发式规范化：

- 清理明显的空白噪声；
- 判断行尾是否缺少终止标点；
- 如果行尾不像完整句，而下一行像续句，则做跨行拼接；
- 如页尾缺少终止标点且下一页开头明显是续句，允许有限度跨页缝合；
- 标题、列表项、明显分页边界不能被粗暴拼接；
- 缝合后仍需保留起止页码元信息。

第一版不追求完美版面恢复，但必须避免“半句成块、语义严重破碎”的情况。

### 明确边界

- 仅支持可提取文本的 PDF；
- 不做 OCR；
- 不承诺复杂表格、双栏、扫描件完美还原；
- 不通过 Tika 处理 PDF。

### 验收

- 文本型 PDF 可稳定建库；
- 查询结果可明确定位 PDF 文件与页码；
- 解析失败的单文件不会中断整轮任务；
- 典型的页内断句与页尾续句不会在切块后大量形成“半句 chunk”。

### 当前状态

- `已完成`
- 已真实落地：
  - `parsers/pdf.py` 已使用 `pypdf` 逐页抽取文本与页码；
  - `normalizers/pdf.py` 已实现空行噪声清理、软换行缝合、有限跨页续句拼接；
  - `extensions/service.py` 已打通 PDF 的预检、全量建库、一次扫描变动更新、删除索引与查询；
  - PDF 索引固定写入 `extensions/pdf/` 独立目录下的 SQLite / 向量实例，不进入 Markdown 主库；
  - 单个 PDF 损坏时只记 warning 并跳过，不会中断整轮任务；
  - 主查询层已通过 broker 融合 PDF 结果，并强制返回 `PDF · 文件名 · 第 N 页` 标签；
  - 已补充 PDF 解析、缝合、独立建库与查询页码的单元测试。

## 最终上线判断标准

只有同时满足以下条件，扩展格式功能才允许被视为可上线：

- 扩展格式关闭时，主链行为零回归；
- Markdown 当前功能没有任何用户感知退化；
- 每条能力线或每个已暴露子格式均可单独建库、单独删除、单独监听；
- 每条能力线或每个已暴露子格式均可单独执行一次扫描变动更新；
- 查询结果带来源身份；
- PDF 结果能说明具体 PDF；
- 任何一种格式坏掉都不会拖垮主链；
- 源路径临时失联不会自动清库；
- 取消勾选目录前一定有明确确认；
- 构建包仍维持轻量策略；
- 自动化测试与人工验收全部通过。

## 一句话总原则

扩展格式功能必须作为一个高度解耦、可独立管理、可独立失败、可独立删除、但查询层可统一融合的平行子系统存在。

它在实现上收敛为：`PDF` 走专门链，`Tika` 走统一扩展链；Tika 支持差的格式灰色可见但不可用；任何实现只要会让当前 Markdown 主链变得更脆、更乱、更难维护，即使它功能上“看起来能跑”，也不得采用。
