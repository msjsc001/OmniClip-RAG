# 扩展格式隔离子系统实施计划

## 文档定位

这份文档是“扩展格式”功能的唯一执行蓝图与持续更新底稿。

后续无论在任何聊天窗、任何开发阶段继续推进扩展格式能力，都必须先阅读本文件，再继续实施。

本文件的目的不是写概念，而是确保以下事情始终成立：

- 新功能在架构上高度解耦；
- 新功能不会污染当前 Markdown 主链；
- 中途换聊天窗、换上下文、甚至上一次对话完全卡死时，也能从这份文档直接接着做；
- 每一阶段做完后，都能把真实进展、真实问题、真实妥协继续写回这里；
- 最终上线时，可用这份文档对照验收，不会“做着做着变形”。

## 当前背景快照

### 当前产品主链

当前软件的稳定主链是：

- 以 Markdown / Logseq 笔记库为核心输入；
- 通过解析、切块、SQLite + FTS5、LanceDB 向量索引完成本地混合检索；
- Qt 为主 UI；
- 打包版为轻量 onedir；
- 运行时数据写入 `%APPDATA%\OmniClip RAG`；
- 当前已重点打磨：
  - 全量建库；
  - 热监听；
  - 中断恢复；
  - OOM / 高压回退；
  - 构建与运行时分离；
  - 版本化打包目录；
  - 双语 UI；
  - 日志与诊断。

### 当前要新增的能力

在现有 Markdown 主链之外，新增一个完全独立的“扩展格式”子系统，第一批优先考虑：

- PDF
- HTML
- MHTML

### 当前已经明确的产品方向

- 扩展格式不是往现有 Markdown 主系统里塞几个 parser。
- 扩展格式要以“隔离子系统”的方式出现。
- 隔离范围包括：
  - UI
  - 配置
  - 数据目录
  - 建库状态
  - 热监听状态
  - 删除/重建
  - 日志标识
- 但会复用现有成熟基建：
  - 任务框架
  - 进度上报
  - 暂停/取消
  - 日志
  - watchdog
  - 预检框架
  - 查询结果融合框架
  - 数据目录总规则
  - UI 样式体系

## 绝对硬约束

以下是不可妥协约束。任何实现只要违反其中任意一条，都视为方案失败。

### 1. 不能破坏当前已有功能

扩展格式功能上线后，即使用户完全不使用它，当前软件的以下行为也必须与现在保持一致：

- Markdown 主建库流程；
- Markdown 热监听；
- 现有查询行为；
- 当前数据目录与索引状态判断；
- 当前打包体积控制策略；
- 当前运行时引导逻辑；
- 当前日志、清理、配置加载；
- 当前 UI 主流程。

执行标准：

- 扩展格式默认关闭；
- 关闭时主链行为必须等价于“此功能不存在”；
- 如出现任何主链行为回归，该阶段不得合入。

### 2. 不能污染主索引状态

扩展格式任何一种格式建库失败、监听失败、配置错误、来源目录失效，都不能影响：

- Markdown 主索引是否存在；
- Markdown 主库是否可查询；
- Markdown 主监听是否正常；
- Markdown 主任务状态显示；
- Markdown 主状态文件。

### 3. 不能删除用户原始文件

无论是删除索引、禁用某格式、移除来源目录，系统都只能操作自身数据目录，不得删除任何用户原始 PDF / HTML / MHTML 文件。

### 4. 打包体积必须受控

扩展格式的第一版设计必须遵守：

- 不能引入 OCR；
- 不能引入浏览器内核；
- 不能引入重量级网页渲染运行时；
- 不能显著膨胀打包版体积；
- 目标仍保持在轻量分发范围内，理想上不超过 `200 MB`。

### 5. 所有扩展格式都必须是“可撤销”的

每种格式都必须支持：

- 启用 / 禁用；
- 建立索引；
- 重建索引；
- 删除该格式索引；
- 停止该格式热监听；
- 清除该格式状态；
- 不影响别的格式和 Markdown 主链。

## 核心设计原则

### 原则 A：物理隔离，逻辑融合

扩展格式底层要“物理隔离”，查询体验可以“逻辑融合”。

含义：

- 各格式各自建库；
- 各格式各自监听；
- 各格式各自状态；
- 查询时再统一聚合结果。

### 原则 B：先做隔离骨架，再接格式解析

禁止一上来就同时把 PDF/HTML/MHTML 全部塞入主链。

正确顺序：

1. 先搭扩展格式子系统骨架；
2. 再接入第一批格式；
3. 再接入独立监听；
4. 最后做查询融合收尾。

### 原则 C：现有成熟能力优先复用

凡是当前主链已经成熟的机制，不重复造轮子：

- Worker
- 任务状态
- 预检
- 暂停/取消
- 日志
- watchdog
- 清理入口
- 打包流程

但所有“状态存储与任务身份”必须做命名空间隔离。

### 原则 D：结果展示必须带来源身份

所有扩展格式结果都必须明确标识来源类型和具体来源名。

例如：

- `Markdown · 项目笔记`
- `PDF · Deep Learning.pdf · 第 12 页`
- `HTML · docs.python.org · asyncio`
- `MHTML · OpenAI 文章归档`

特别要求：

- PDF 结果不能只显示 `PDF`，必须至少显示 PDF 文件名；
- 如果能够可靠拿到页码，则必须显示页码；
- 查询融合后也必须保留该来源身份。

- PDF
- HTML
- MHTML
- 独立来源目录管理

## UI 方案

### 位置

在“配置”中新增独立页签：

- `扩展格式`

### 顶部固定区域

顶部固定区不随子页签变化，统一保留：

1. 页头说明
2. 状态总览
3. 全局操作

### 顶部说明文案方向

建议文案方向：

> 这里用于管理 PDF / HTML / MHTML 等扩展资料格式。它们独立预检、独立建库、独立删除、独立监听，但查询时可以统一融合。

### 状态总览

状态总览样式应与“开始”页的状态芯片风格一致。

每个格式一个芯片，例如：

- `PDF：未启用`
- `PDF：未配置`
- `PDF：未建库`
- `PDF：建库中`
- `PDF：监听中`
- `PDF：已建立`
- `PDF：需要重建`
- `PDF：出错`

总览区域还应额外给出汇总：

- `已启用 2 项 / 已建立 1 项`

### 全局操作

全局操作统一保留在顶部：

- `预检建库时间空间`
- `建立全部已启用格式索引`
- `停止当前扩展任务`

说明：

- 按用户要求，原先讨论中的 `扫描已启用格式 / 扫描文件` 统一改为 `预检建库时间空间`；
- 该语义与当前主链一致；
- 不再额外造一个“扫描文件”概念。

### 子页签结构

扩展格式内部使用页签分页：

- `PDF`
- `HTML`
- `MHTML`

这样比把所有格式卡片堆在一个长页面里更清晰。

### 每个格式页签内的内容

每个格式页签都保持同构，字段尽量一致：

| 区块 | 字段 |
| --- | --- |
| 基本状态 | 启用开关 |
| 基本状态 | 状态芯片 |
| 来源配置 | 来源目录列表（支持多选） |
| 来源配置 | 添加目录 / 移除目录 |
| 来源配置 | 勾选当前工作区关联目录 |
| 统计信息 | 文件数 |
| 统计信息 | 索引体积 |
| 统计信息 | 上次建库时间 |
| 统计信息 | 热监听状态 |
| 统计信息 | 当前说明 |
| 操作区 | 预检建库时间空间 |
| 操作区 | 建立索引 |
| 操作区 | 重建索引 |
| 操作区 | 删除索引 |
| 操作区 | 打开数据目录 |
| 操作区 | 启动热监听 / 停止热监听 |

### 来源目录管理规则

来源目录必须支持多选，但要守住隔离边界。

允许两类来源：

1. 当前工作区关联目录
2. 额外自定义目录

额外规则：

- 如果某个“内置可勾选目录”已经从当前工作区移除，则扩展格式里的关联项也必须同步移除；
- 不允许直接引用其他工作区的目录配置；
- 不在第一版支持跨工作区联合来源。

## 数据与状态隔离设计

### 数据目录结构

建议目录结构如下：

```text
%APPDATA%\OmniClip RAG\
└─ workspaces\
   └─ <workspace-id>\
      ├─ markdown\
      │  ├─ sqlite\
      │  ├─ state\
      │  └─ exports\
      └─ extensions\
         ├─ pdf\
         │  ├─ config.json
         │  ├─ state\
         │  ├─ sqlite\
         │  ├─ vector\
         │  ├─ logs\
         │  └─ exports\
         ├─ html\
         │  ├─ ...
         └─ mhtml\
            ├─ ...
```

### 每种格式独立状态文件

每种格式至少独立维护：

- `config`
- `index_state`
- `build_state`
- `watch_state`
- `stats`
- `preflight_history`

禁止行为：

- 禁止扩展格式与 Markdown 共用同一份 `index_state.json`
- 禁止扩展格式共用 Markdown 的 `watch_state.json`
- 禁止扩展格式把自身结果写入主 Markdown 索引目录

### 删除语义

删除动作必须拆成三种，不得混淆：

1. `禁用该格式`
   - 不参与查询
   - 不删除索引

2. `删除该格式索引`
   - 只删除该格式在 AppData 下的索引与状态
   - 不删除原始文件

3. `移除该格式来源配置`
   - 只移除目录配置
   - 不删除原始文件

## 代码层隔离设计

### 命名空间建议

新增独立子系统命名空间：

```text
omniclip_rag/extensions/
  ├─ models.py
  ├─ registry.py
  ├─ service.py
  ├─ query.py
  ├─ watch.py
  ├─ preflight.py
  ├─ progress.py
  └─ parsers/
     ├─ pdf.py
     ├─ html.py
     └─ mhtml.py
```

### 模块职责

#### `registry.py`

负责：

- 维护启用状态；
- 管理来源目录；
- 汇总状态；
- 向 UI 提供总览摘要。

#### `service.py`

负责：

- 单格式建库；
- 单格式重建；
- 单格式删除；
- 单格式状态恢复。

#### `watch.py`

负责：

- 单格式独立热监听；
- 监听状态落盘；
- 监听恢复；
- 不同格式监听任务的隔离。

#### `query.py`

负责：

- 从不同来源索引拉取结果；
- 合并结果；
- 写入来源身份；
- 给主查询层返回统一结构。

#### `preflight.py`

负责：

- 对单格式或全部已启用格式做预检；
- 复用现有预检思路；
- 但计算来源目录、文件数、体积、估时时必须按格式隔离。

#### `parsers/*`

负责各格式解析，统一产出中间结构，不得直接侵入主链存储逻辑。

### 必须复用但不得污染的现有能力

以下能力应复用：

- Worker/任务框架
- 日志
- 任务进度快照
- 暂停 / 取消
- watchdog
- 资源控制器
- 预检展示样式
- 状态芯片样式

但扩展格式的任务 ID、状态文件、存储命名空间必须独立。

## 查询融合设计

### 目标

查询体验统一，但底层索引物理隔离。

### 设计方式

采用 broker 模式：

1. Markdown 查询服务
2. PDF 查询服务
3. HTML 查询服务
4. MHTML 查询服务

由上层统一调度，并做最终融合。

### 结果结构要求

每条结果必须带：

- `source_kind`
  - `markdown`
  - `pdf`
  - `html`
  - `mhtml`
- `source_label`
  - 例如 `PDF · Deep Learning.pdf · 第 12 页`
- `source_name`
  - 原文件名或来源标题
- `source_path`
- `page_no`（如可用）
- `section_heading`（如可用）

### 查询页过滤建议

建议增加来源过滤：

- `全部`
- `Markdown`
- `PDF`
- `HTML`
- `MHTML`

或多选版本：

- `[√] Markdown`
- `[√] PDF`
- `[ ] HTML`
- `[ ] MHTML`

### 统一排序要求

即使不同来源融合，排序也必须继续使用统一可解释的最终分数逻辑。

禁止行为：

- 仅因为结果来自 PDF 就先天降低或抬高权重；
- 仅因为来源类型不同而做不可解释排序。
- 直接把不同 LanceDB 表返回的原始向量距离分数混在一起做全局排序。

### 多表向量分数对齐原则

扩展格式采用物理隔离后，Markdown、PDF、HTML、MHTML 会各自拥有独立的向量表。不同表的数据分布不同，原始 distance / similarity 分数不能直接横向比较。

因此跨格式查询必须采用“两段式融合”，而不是“多表 raw score 直接拼起来排序”：

1. 每个来源内部先独立召回并形成本来源候选；
2. 每个来源内部先完成本地排序；
3. 跨来源融合时优先依赖以下信号，而不是直接使用跨表 raw distance：
   - 来源内 rank
   - 来源内归一化分数（percentile / min-max / calibration window）
   - SQLite / FTS5 字面命中分数
   - 结构化命中信号（标题、页码、章节命中等）
   - 可选 reranker 的统一结果

第一版建议采用稳妥策略：

- 向量 raw score 只在“单来源内部”参与排序；
- broker 融合时以 rank-based fusion 为主；
- FTS5 / lexical score 作为跨来源校验信号；
- 如果启用了 reranker，则 reranker 作为最终统一裁决层。

这条规则是硬约束，不是优化项。任何跨来源融合如果直接比较不同表的原始距离分数，都视为实现错误。

## 热监听设计

### 总原则

热监听必须独立。

每一种格式的监听：

- 独立开关
- 独立状态
- 独立日志
- 独立恢复
- 独立报错

### 复用要求

应尽量复用现有成熟热监听框架，包括：

- 稳定窗口
- 快照 diff
- 删除确认
- 失败回退
- watch state 持久化

### 风险控制

第一版虽然支持扩展格式热监听，但必须限制重任务并发：

- Markdown 全量建库
- PDF 全量建库
- HTML 全量建库
- MHTML 全量建库

这些属于重任务，应由全局任务协调器做串行或受控调度，避免资源争抢。

扩展格式监听不得无脑在重建库过程中同时启动大规模重建。

## 包体与依赖约束

### 体积控制目标

扩展格式加入后，打包版仍应尽量维持轻量，目标不超过 `200 MB`。

### 第一版依赖选择原则

#### PDF

优先考虑：

- `pypdf`

禁止：

- OCR 引擎
- 图像识别库
- 扫描件识别栈

#### HTML / MHTML

优先考虑：

- Python 标准库
- `beautifulsoup4`
- 轻量 HTML 解析

禁止：

- `QtWebEngine`
- `Playwright`
- `Chromium`
- 浏览器运行时
- JS 执行环境

### 打包红线

- 不得把额外大模型打包进去；
- 不得把文档转换重依赖拉入主包；
- 不得因为支持扩展格式而破坏当前轻量发布策略。

## 致命风险清单与防护

### 风险 1：主链被污染

#### 表现

- 扩展格式失败后主 Markdown 显示“索引损坏”；
- 主查询异常；
- 主监听状态错乱。

#### 防护

- 独立状态文件；
- 独立索引目录；
- 关闭时行为等价测试；
- 所有扩展入口默认关闭。

### 风险 2：多重任务争抢导致资源崩溃

#### 表现

- Markdown 建库与 PDF 建库同时跑；
- RAM/VRAM 被同时拉爆；
- 用户误以为系统坏掉。

#### 防护

- 建立全局任务协调器；
- 重任务串行；
- 监听回压；
- 保留现有 watchdog / 降批 / 诊断。

### 风险 3：来源目录重复导致重复索引

#### 表现

- 同一 PDF 被两个来源目录同时命中；
- 查询中重复出现；
- 索引膨胀。

#### 防护

- 规范化路径去重；
- 文件指纹去重；
- 来源映射表。

### 风险 4：单文件解析失败中断整轮任务

#### 表现

- 某个脏 PDF 直接把全部扩展建库打断。

#### 防护

- 单文件失败只记日志并跳过；
- 整轮任务继续；
- 失败计数出现在状态摘要里。

### 风险 5：MHTML / HTML 带恶意脚本或脏内容

#### 防护

- 只做文本抽取；
- 不执行脚本；
- 不加载外部资源；
- 不做浏览器渲染。

### 风险 6：跨表向量分数失真导致融合排序不公

#### 表现

- Markdown、PDF、HTML 的向量距离分数被直接混排；
- 某个来源因为自身表分布更集中而长期霸榜；
- 查询结果看起来像“来源偏置”，而不是相关性偏置。

#### 防护

- 不允许跨表直接比较 raw distance；
- 必须先做来源内排序，再做跨来源 rank-based fusion；
- FTS5 / lexical score 作为跨来源二次校验；
- 如启用 reranker，则以 reranker 作为最终统一裁决层。

### 风险 7：PDF 物理行抽取导致语义碎裂

#### 表现

- 一句话在页尾断开，在下一页续上；
- pypdf 按物理行输出后直接切块，导致 chunk 里全是半句；
- 查询命中虽然有字面重合，但语义向量质量极差。

#### 防护

- PDF 必须先经过“文本规范化 + 伪断句缝合”层，再进入切块；
- 缝合逻辑至少要处理行内断句、跨行续接、必要的跨页续接；
- 仍要保留页码跨度元信息，避免缝合后丢失定位能力。

### 风险 8：聊天窗切换导致上下文丢失

#### 防护

- 本文档必须持续更新；
- 每完成一个阶段，必须把真实状态写回本文件；
- 新聊天窗继续时，以本文件为唯一权威背景。

## 阶段 3：PDF 解析与独立建库

### 目标

支持文本型 PDF。

### 交付

- PDF 文本抽取；
- 页码元信息；
- PDF 文本规范化层；
- 伪断句缝合逻辑；
- 独立预检；
- 独立建库 / 重建 / 删除；
- 查询展示为 `PDF · 文件名 · 第 N 页`

### PDF 解析硬约束

PDF 不能按物理行原样直接切块。必须在切块前做一次轻量启发式规范化：

- 清理明显的空白噪声；
- 判断行尾是否缺少终止标点；
- 如果行尾不像完整句，而下一行像续句，则做跨行拼接；
- 如页尾缺少终止标点且下一页开头明显是续句，允许有限度跨页缝合；
- 标题、列表项、明显分页边界不能被粗暴拼接；
- 缝合后仍需保留起止页码元信息。

第一版不追求完美版面恢复，但必须避免“半句成块、语义严重破碎”的情况。

### 明确边界

- 仅支持可提取文本的 PDF；
- 不做 OCR；
- 不承诺复杂表格、双栏、扫描件完美还原。

### 验收

- 文本型 PDF 可稳定建库；
- 查询结果可明确定位 PDF 文件与页码；
- 解析失败的单文件不会中断整轮任务；
- 典型的页内断句与页尾续句不会在切块后大量形成“半句 chunk”。

## 测试与验收门槛

## 自动化测试必须覆盖

### 主链不回归

- 扩展格式关闭时主链行为不变；
- 现有测试全部继续通过；
- 扩展格式配置存在时，Markdown 主建库与查询仍正确。

### 隔离性

- 删除 PDF 索引不影响 HTML / MHTML / Markdown；
- HTML 建库失败不污染主索引状态；
- MHTML 监听崩溃不影响其他来源。

### 查询融合

- 结果带来源标签；
- PDF 标签带文件名；
- 多来源混合排序稳定；
- 来源过滤正确。

### 多来源目录

- 目录去重；
- 来源移除同步；
- 当前工作区关联目录变更同步。

### 打包

- 包体不突破目标阈值；
- 不额外打入重量级依赖；
- 启动速度无明显恶化。

## 最终上线判断标准

只有同时满足以下条件，扩展格式功能才允许被视为可上线：

- 扩展格式关闭时，主链行为零回归；
- Markdown 当前功能没有任何用户感知退化；
- 每种格式均可单独建库、单独删除、单独监听；
- 查询结果带来源身份；
- PDF 结果能说明具体 PDF；
- 任何一种格式坏掉都不会拖垮主链；
- 构建包仍维持轻量策略；
- 自动化测试与人工验收全部通过。

## 一句话总原则

扩展格式功能必须作为一个高度解耦、可独立管理、可独立失败、可独立删除、但查询层可统一融合的平行子系统存在。

如果某项实现会让当前 Markdown 主链变得更脆、更乱、更难维护，即使它功能上“看起来能跑”，也不得采用。
