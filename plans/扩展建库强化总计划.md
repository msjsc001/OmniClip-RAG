# 扩展建库强化总计划

## 文档定位

这份文档是 `OmniClip RAG` 后续推进“扩展格式建库强化”的唯一主计划与接力底稿。

它要解决的不是一个零碎 Bug，而是一整条扩展建库链当前明显落后于 Markdown 主线的问题，包括：

- `PDF` 扩展出现“几乎全跳过”的实机故障；
- `Tika` / `PDF` 扩展建库进度不完整、不可观察、容易让用户误判为卡死；
- 扩展建库几乎没有主线同级别的暂停、取消、中断感知、恢复与诊断能力；
- 扩展打包链、依赖链、状态链、UI 任务链之间没有形成像主线那样成熟的闭环。

这份文档必须满足 4 个目标：

1. 后续无论换哪个聊天窗、换哪个 AI、换哪个开发阶段，都能直接接着做；
2. 后续执行者不需要重新从头梳理“到底哪里坏了、为什么不能直接硬搬主线”；
3. 后续执行者能明确知道先做什么、后做什么、哪些绝对不能先做；
4. 后续真正开始改代码时，可以把本文件作为实施蓝图、验收清单与风险规约一起使用。

## 当前真实背景

### 1. 产品主链与扩展链的关系

当前项目已经不是一个单一 Markdown 索引器，而是以下几条链路共同组成：

- Markdown / Logseq 主建库与主查询
- 本地向量与重排模型管理
- Runtime 侧车与打包运行时
- 只读 MCP Server
- 扩展格式子系统：
  - `PDF`：独立 parser、独立扩展存储、独立扩展向量表
  - `Tika`：外置 sidecar + 多格式扩展存储 / 向量表

扩展子系统当前是**物理隔离命名空间**，不会与 Markdown 主索引混写。这是一个非常重要的前提，因为它决定了：

- 扩展的问题当前主要局限在扩展子系统内部；
- 后续强化时不应该轻易把扩展与主线强行统一成一套数据流；
- 但“控制面”能力完全有必要向主线看齐。

### 2. 本轮对话中已经查实的扩展问题

这次对话里已经查实，而不是猜测的事实如下。

#### 2.1 PDF 扩展当前故障不是“目录选错”，而是解析链本身坏了

实机日志里已多次出现：

- `ModuleNotFoundError: No module named 'omniclip_rag.extensions.parsers.pdf'`

这说明打包版在运行时尝试动态导入 [pdf.py](/D:/软件编写/OmniClip%20RAG/omniclip_rag/extensions/parsers/pdf.py) 时，包里没有把它带进去。

同时，源码态本地导入 [pdf.py](/D:/软件编写/OmniClip%20RAG/omniclip_rag/extensions/parsers/pdf.py) 还暴露出第二层问题：

- `ModuleNotFoundError: No module named 'pypdf'`

也就是说，`PDF` 路线目前是“两层一起坏”：

1. 打包产物漏了 `omniclip_rag.extensions.parsers.pdf`
2. 源码环境还缺 `pypdf`

所以用户看到的“PDF 建库几乎全跳过”不是误操作，也不是目录问题，而是解析链本身未闭环。

#### 2.2 Tika 没有出现与 PDF 同类的 parser 漏包问题

已查实：

- `Tika` parser 在打包分析中存在
- 运行日志里也出现过：
  - `Tika sidecar is healthy on port 9998`

因此当前判断是：

- `Tika` 当前没有和 `PDF` 一样的“动态导入模块漏包”问题
- 但 `Tika` 的建库体验与中断保护仍明显不足

#### 2.3 扩展建库的“进度”当前不是全流程进度

已查实：

- 扩展页全局建库主要走 `FunctionWorker`
- 只有按单个来源目录做扫描 / 建库时，才走 `ProgressFunctionWorker`
- 即便是行级别建库，后端也只发粗阶段：
  - `scan_sources`
  - `parse_tika` / `parse_pdf`
  - `write_vector`
  - `finalizing`

当前缺失的可见过程包括：

- `Tika runtime` 启动过程
- 单个大文件内部解析过程
- 向量写入内部进度
- 全局扩展建库的真正实时进度

所以用户观察到“进度停滞、像卡死、看不出是否还在工作”，这是符合代码现状的真实现象。

#### 2.4 扩展建库没有主线那种中断保护

已查实：

- 主线 Markdown 建库具备：
  - `pause_event`
  - `cancel_event`
  - `BuildCancelledError`
  - 持久化 `rebuild state`
  - resume 入口
  - watchdog 卡死诊断
- 扩展 PDF / Tika 建库目前：
  - 没有同级别的 `pause / cancel`
  - 没有同级别的 checkpoint / resume
  - 没有同级别的 watchdog
  - 没有下次启动后的恢复提示

当前扩展最多只有两种“弱保护”：

- 任务互斥协调器，避免扩展任务与主线任务硬撞
- 扩展独立命名空间，避免直接污染 Markdown 主索引

这两者都不是主线级的“中断保护”。

#### 2.5 扩展 full rebuild 当前是“先清后建”

已查实：

- `PDF` full rebuild 与 `Tika` full rebuild 都存在：
  - `self.store.reset_all()`
  - `self.vector_index.reset()`
  - 然后再重新扫描 / 解析 / 写入

这意味着：

- 一旦中途退出、崩溃、断电、强关
- 当前扩展索引非常容易停在半成品状态
- 且不会像主线一样自然续建

## 为什么这次改造必须做

### 产品层理由

扩展格式已经不是“可有可无的小实验”。

用户现在已经真实点到并依赖：

- `PDF` 建库
- `Tika` 大格式目录建库
- 扩展来源目录管理
- 扩展独立查询与状态反馈

如果扩展链长期停留在：

- parser / 依赖不闭环
- 进度不可见
- 中断后要靠运气
- 打包态和源码态不一致

那它对用户来说就不是“半成熟功能”，而是会持续消耗信任的“看起来有，但不敢真用”的功能。

### 工程层理由

当前主线 Markdown 建库已经积累出一套成熟的控制面能力：

- 进度协议
- 暂停 / 取消
- checkpoint / resume
- watchdog
- ETA
- UI 协调

如果扩展线长期不接入这些成熟基建，会形成两套产品体验：

- 主线像工业级流程
- 扩展像试验性后台脚本

这会让整个产品越来越割裂。

## 本次改造的最终目标

“扩展建库强化”最终要达到的不是一句“修复几个 Bug”，而是以下完整结果。

### 目标 1：扩展解析链闭环

- `PDF` 在源码态与打包态都能稳定解析
- `Tika` 在源码态与打包态都能稳定运行 sidecar 并建库
- 动态导入、可选依赖、打包隐藏导入都要统一闭环

### 目标 2：扩展建库全过程可观察

用户必须能在界面或日志中看懂：

- 当前是不是已经开始
- 现在处于哪个阶段
- 当前正在处理哪个来源目录 / 哪个文件
- 当前是否还在前进
- 哪些文件被跳过，哪些是真失败

### 目标 3：扩展具备至少接近主线的中断控制

最少要具备：

- 可取消
- 可识别异常中断
- 可提示用户“上次中断”
- 可安全重来

较完整目标则包括：

- 可暂停
- 可恢复
- 有 checkpoint
- 有 watchdog

### 目标 4：扩展强化不能反噬主线

扩展的任何强化都必须满足：

- 不破坏 Markdown 主建库
- 不破坏 Markdown 主监听
- 不破坏主查询
- 不破坏 Runtime / 打包 / MCP 现有稳定线

### 目标 5：扩展子系统仍然保持隔离

强化之后，扩展仍然要保持：

- 独立来源目录
- 独立状态
- 独立索引
- 独立删除 / 重建 / 清理

不能为了“复用主线能力”而把扩展强塞进 Markdown 主索引。

## 非目标

这次改造里**明确不应被当作优先目标**的事项如下。

### 1. 不是要把扩展和主线重构成一套完全统一的数据流

不做：

- 让 `PDF / Tika` 直接写进 Markdown 主存储
- 强行复用 Markdown parser / rendering / dependency closure 流程

原因：

- 扩展数据面与主线数据面语义不同
- 这样做会不必要地扩大回归面

### 2. 不是优先做“原子切换式建库”

“原子切换式建库”本身不是坏事，但当前扩展最大问题不是它。

当前真正急迫的是：

- parser / 依赖闭环
- 进度可见
- 中断保护
- 恢复语义

扩展当前本来就处于独立命名空间，因此原子切换不应成为第一优先。

### 3. 不是一上来就做极复杂的 chunk 级 / vector 级恢复

第一版绝对不应追求：

- chunk 级 resume
- 向量写入中间段精细续传
- 多阶段 cursor 极细恢复

原因：

- 这会把一致性风险拉到最高
- 最容易出现“UI 显示成功、实际索引残缺”的隐性故障

---

## 核心架构判断

这次改造最重要的架构判断只有一句：

> **扩展建库应该复用主线的“控制面”，而不是复用主线的“数据面”。**

### 什么是“控制面”

这里的“控制面”指：

- `pause_event / cancel_event`
- 统一 progress payload
- checkpoint / state persistence
- resume prompt
- watchdog
- 任务状态机
- UI 与 worker 的协作协议

这些能力本质上与“处理 Markdown 还是 PDF/Tika”关系不大，属于通用的建库控制能力。

### 什么是“数据面”

这里的“数据面”指：

- parser
- manifest / file enumeration
- Tika runtime 启动
- PDF / Tika 文件解析
- 扩展 store 与扩展 vector index 的实际写入方式
- 扩展来源目录、格式选择与删除策略

这些能力必须保留扩展自己的实现。

### 结论

最好的改造方式不是二选一：

- 不是“完全独立再造一套”
- 也不是“把主线整套硬接过来”

而是：

> **先复制 / 下沉主线成熟的控制面能力，再让扩展的 PDF/Tika 数据流挂接上去。**

这条路线风险最低，也最符合当前项目已经形成的边界。

## 必须先写死的设计合同

以下不是“实现时顺手考虑”的软性建议，而是后续真正动代码前必须先写死的合同。

如果这些合同不先明确，后面最容易出现的不是普通小 Bug，而是：

- 假 `READY`
- 假 `CANCEL`
- 假 `RESUME`
- 索引内容悄悄残缺
- UI、状态文件、查询结果三套语义互相打架

### 1. 扩展状态机真相表

必须先定义扩展建库状态机，并明确每个状态的：

- 进入条件
- 退出条件
- 持久化字段
- UI 文案
- 查询可用性
- 可执行动作

最低至少要覆盖：

- `DISABLED`
- `NOT_BUILT`
- `BUILDING`
- `PAUSED`
- `CANCELLING`
- `INTERRUPTED`
- `RESUMABLE`
- `READY`
- `STALE`
- `ERROR`
- `WATCHING`

### 2. `READY` 与完成判定合同

必须明确写死：什么情况下扩展索引才允许进入 `READY`。

最低完成判定应包含：

- store 中的文件 / chunk 统计与当前 build 结果一致
- 向量文档数与可向量化文件集合一致，或明确进入“仅 lexical ready”合同
- 当前 build 没有处于 `interrupted / cancelling / paused`
- 本次 build 的关键失败数没有越过不可接受阈值
- 当前 pipeline 的状态文件与 UI 刷新已同步完成

### 3. `last-good` 旧索引服务合同

必须明确：

- 扩展 full rebuild 过程中，查询是否允许继续使用上一次成功完成的旧索引
- 若允许，什么条件下仍允许服务查询
- 若不允许，UI 和查询层必须如何诚实降级

### 4. 扩展降级查询合同

必须明确扩展是否允许：

- store 已写好，但 vector 阶段失败时，以 lexical-only 方式继续提供扩展查询
- `Tika / PDF` 在向量 runtime 不可用时，退化为仅字面 / 结构化检索

### 5. build-watch handoff 合同

必须明确：

- 扩展建库过程中，来源目录又发生新增 / 修改 / 删除时怎么办
- 这些变化是记入 `pending changes`，还是要求重扫，还是建完后再补增量
- build 完成后，watch 如何接管 build 期间积累的变化

### 6. durable commit 与恢复单元合同

必须先定义“一个文件什么时候算真正完成，可用于恢复”。

首版建议明确为：

- 单文件 parser 已完成
- 单文件 store 写入完成
- 对应的 durable 状态已刷新
- 若 vector 采用保守重建策略，则文件级 durable commit 以 store 完成为准

### 7. 文件级恢复的 fingerprint 合同

文件级恢复绝不能只依赖路径。

至少要绑定：

- 规范化后的相对路径
- 文件大小
- 修改时间
- 最好再带轻量 fingerprint / 内容签名

### 8. 任务冲突矩阵

必须明确这些组合的行为：

- `build vs watch`
- `build vs delete-index`
- `build vs source-toggle`
- `build vs app-exit`
- `build vs runtime-repair`
- `build vs workspace-switch`
- `build vs data-root-switch`
- `build vs second process`

### 9. 跨进程锁与 lease 合同

当前扩展协调器主要是单进程内互斥。

但真实用户可能：

- 开两个 GUI 实例
- GUI 与别的 headless 入口并发
- 在同一 `data_root` 上重复启动

因此必须定义扩展 build 的跨进程锁 / lease 规则。

### 10. 状态 / schema 版本化与迁移合同

必须明确以下文件都要带版本：

- 扩展 build state
- 扩展 registry / snapshot
- 扩展 store schema
- 扩展 vector 元数据
- watchdog / diagnostics 元数据

并明确：

- 读到旧版本时如何迁移
- 无法迁移时如何回退成“清理状态 + 重建”
- 状态文件写入必须使用原子落盘，不能半写

### 11. 索引失效触发表

必须明确哪些变化会强制扩展索引失效并要求全量重建。

至少包括：

- parser 版本变化
- chunk 规则变化
- `Tika` 提取策略变化
- store schema 变化
- vector 文档布局变化
- 扩展索引版本变化
- 第三方解析栈版本变化（如 `pypdf / Tika / Java`）

### 12. 发布保险与回退合同

必须考虑：如果新恢复链上线后暴露重大问题，如何快速退回安全模式。

建议计划里保留：

- feature flag / capability flag
- 允许临时关闭 resume，仅保留 cancel + interrupted 感知
- 允许在发现问题时回退到“扩展索引需重建”的保守模式

### 13. 抽取结果膨胀上限合同

扩展链面对的不只是“解析失败”，还包括“解析成功但抽出异常大的脏文本”。

因此必须明确：

- 单文件最大正文长度
- 单文件最大 chunk 数
- 单文件最大页数 / 片段数保护阈值
- 超阈值后的处理方式：
  - 截断
  - 标记为异常跳过
  - 记录为 failed / quarantined

否则极端文件会把：

- store
- vector
- UI
- 恢复状态

一起拖垮。

### 14. 扩展文档 ID / chunk ID 稳定性合同

后续如果要安全支持：

- delete-index
- rebuild
- resume
- last-good
- 查询去重

就必须保证扩展文档与 chunk 的 ID 生成规则是稳定且可解释的。

因此必须明确：

- 文档 ID 如何生成
- chunk ID 如何生成
- 在相同输入与相同规则下，ID 必须稳定
- 规则变化时必须明确触发索引失效或重建

否则后面最容易出现：

- 删除删不干净
- 恢复重复写
- 查询结果重复或漏删

### 15. 第三方解析栈版本钉住与漂移控制合同

扩展建库高度依赖第三方解析栈。

至少要明确：

- `pypdf`
- `Tika Server`
- `Java`

这几类依赖的版本策略。

必须规定：

- 哪些版本需要 pin
- 升级这些版本时，需要做哪些 smoke / 回归
- 哪些升级应视为“解析行为变化”，从而触发扩展索引失效或强制重建

否则同一个文件今天和明天可能抽出不同结果，进而破坏：

- build 结果一致性
- resume 可用性
- 打包态与源码态的一致性

---

## 绝对硬约束

以下约束必须视为不可妥协边界。任何实现只要触碰其中任意一条，都应判定为方案失败。

### 1. 不允许破坏 Markdown 主线

扩展强化期间，以下能力不得回归：

- 主线全量建库
- 主线监听
- 主线查询
- 主线向量 / 重排
- Runtime 管理
- MCP

### 2. 不允许为了“统一”而破坏扩展隔离

不允许：

- 让扩展共享 Markdown 主索引表
- 让扩展状态写进 Markdown 主状态文件
- 让扩展清理误删主线数据

### 3. 不允许在“恢复能力”尚未设计清楚前上复杂 resume

绝对禁止：

- 第一步就做 chunk 级 resume
- 第一步就做 vector 阶段精细续建
- 没有 manifest / source 配对校验就直接恢复旧 cursor

### 4. 不允许只在源码态验证，不验证打包态

每一阶段只要涉及：

- parser
- 可选依赖
- Tika runtime
- hidden import
- frozen worker

就必须同时验证：

- 源码态
- onedir / EXE 态

### 5. 不允许让 UI 状态先于真实数据状态“撒谎”

绝对禁止：

- 真实索引未就绪，但 UI 显示 `ready`
- 构建中断后 UI 仍显示成功完成
- 恢复不可用时却展示“可恢复”

### 6. 扩展文档输入必须视为不可信

`PDF / Tika` 面对的是现实世界里的任意文件。

因此必须默认这些输入可能：

- 损坏
- 超大
- 恶意构造
- 让 parser / sidecar 卡住
- 触发超高内存占用

后续实现必须遵守：

- 单文件解析必须有超时或可回收策略
- `Tika` sidecar 请求必须有超时与 recycle 机制
- 任何扩展解析失败都不得把主进程拖死

### 7. 所有 build / 状态文件必须原子写入

扩展强化后会新增：

- build state
- resume 元数据
- lease / lock 元数据
- watchdog 报告

这些文件都必须原子写入，避免程序崩溃时留下半写状态。

### 8. 扩展强化不得引入无法清理的脏数据

所有新引入的扩展状态、恢复文件、lease 文件、watchdog 报告，都必须：

- 放在受控数据目录中
- 有明确命名规则
- 有清理入口或生命周期规则
- 不允许越积越多、最后只能手删

---

## 推荐的总体实施路线

### 总策略

采用“先止血、再可见、再可停、再可恢复”的四段式改造路线：

1. 修确定性故障
2. 补观测能力
3. 补控制能力
4. 补恢复能力

这比“一次性把扩展重构成主线镜像版”更稳得多。

---

## 阶段 0：实施前准备

### 目标

在真正动扩展建库代码之前，先把“现状基线”和“未来验证口径”固定下来。

### 要做什么

1. 固定扩展专项验收样本
2. 固定扩展专项日志观察点
3. 固定扩展专项测试入口
4. 固定打包态回归入口
5. 固定扩展状态机真相表
6. 固定任务冲突矩阵
7. 固定索引失效触发表
8. 固定发布门禁与自救路径

### 推荐样本集

按格式准备最小、可重复、可控的样本集：

- `PDF`
  - 正常文本 PDF
  - 含中文 PDF
  - 空 PDF / 损坏 PDF
  - 超大 PDF
- `Tika`
  - `EPUB`
  - `DOCX`
  - `TXT`
  - 一个空文件
  - 一个 sidecar 可识别但无正文的文件

### 交付物

- 一组稳定样本
- 一份专项测试清单
- 一份“源码态 / 打包态都要跑”的验证脚本说明
- 一份扩展状态机真相表
- 一份任务冲突矩阵
- 一份索引失效触发表
- 一份发布门禁清单

### 风险

如果没有先固定样本，后面每次修复都容易变成“看起来好了，但没有真正可比口径”。

如果没有先固定状态机、冲突矩阵与失效触发表，后面最容易出现“代码写得很快，但每个人理解的状态语义都不一样”。

## 阶段 1：修复解析链与打包依赖闭环

### 目标

先让扩展能稳定“建起来”，而不是一开始就碰复杂的控制语义。

### 范围

- `PDF` parser hidden import 闭环
- `pypdf` 依赖闭环
- `Tika` parser / sidecar 入口保持可打包
- 扩展 parser 动态导入治理
- 第三方解析栈版本组合钉住

### 推荐实现方式

1. 明确把扩展 parser 视为打包显式依赖
2. 对动态导入模块建立稳定 hidden import 列表
3. 对 `pypdf` 做打包态与源码态共同校验
4. 在扩展自检里给出更明确的 parser / dependency 失败原因
5. 明确 `PDF / Tika` 解析链面对不可信文件时的超时、内存与失败隔离策略
6. 钉住并验证 `pypdf / Tika Server / Java` 的受支持版本组合

### 为什么先做这一层

因为它是**确定性问题**：

- 风险低
- 收益高
- 不需要先碰状态机

### 验收标准

- `PDF` 不再因 `ModuleNotFoundError: omniclip_rag.extensions.parsers.pdf` 全跳过
- 源码态与 EXE 态都能解析最小 PDF 样本
- `Tika` 路线保持可用且不回归

### 绝对不要做的事

- 不要在这一阶段顺手引入 resume
- 不要在这一阶段顺手重构扩展状态机
- 不要把 parser 修复和 UI 大改绑在一起

---

## 阶段 2：补齐扩展建库的观测能力

### 目标

让用户能“看懂扩展建库在做什么”，避免“还在工作但看起来像死了”。

### 范围

- 全局扩展建库改用 progress worker
- 统一扩展进度 payload
- 统一扩展阶段文案
- 统一日志与状态栏反馈

### 推荐实现方式

#### 2.1 全局扩展任务从 `FunctionWorker` 升级为 progress task

当前扩展全局建库几乎没有实时 progress。

建议：

- 让全局扩展重建与预检也走 progress worker
- 把全局任务和行级任务统一到同一套 payload 协议

#### 2.2 扩展统一进度协议

建议扩展进度协议至少包含：

- `pipeline`: `pdf` / `tika`
- `stage`
- `stage_status`
- `current`
- `total`
- `overall_percent`
- `current_path`
- `current_file`
- `skipped_count`
- `failed_count`
- `eta_seconds`
- `close_safe`
- `build_id`

#### 2.3 扩展阶段应覆盖全流程

推荐至少拆成：

- `prepare_runtime`
- `scan_sources`
- `parse_files`
- `write_store`
- `write_vector`
- `finalizing`

#### 2.4 加心跳与 stall 可见性

在无法拿到真实细粒度进度时，也至少要让用户看到：

- “当前仍在处理”
- “最近 x 秒仍有前进 / 暂无前进”
- 当前目录大小 / 文件数是否变化

#### 2.5 统一 `build_id` 与日志关联

扩展建库后续一旦引入：

- 心跳
- watchdog
- resume
- 重建后查询

就必须保证所有日志、状态与诊断都能追溯到同一轮 build。

因此建议从一开始就统一：

- `build_id`
- `workspace id`
- `pipeline`
- `data_root`

并让：

- 进度 payload
- build state
- watchdog 报告
- UI 日志

都带上同一个 `build_id`。

#### 2.6 日志节流与 UI 刷新节流

心跳、阶段更新、单文件状态都增强后，必须同步考虑：

- 日志洪泛
- UI 频繁刷新
- 大量文件时日志可读性迅速下降

因此需要提前规定：

- 心跳周期
- 大批量文件时的采样策略
- UI 刷新节流阈值

### 验收标准

- 用户能看清当前处于哪个阶段
- 大文件、慢 sidecar、慢向量写入时，不再只停在一条静止提示上
- 全局建库和单目录建库的进度体验明显收拢

### 风险

这一阶段最容易出现的问题不是崩，而是：

- payload 字段不统一
- UI 层与后端阶段文案不一致
- 日志与 UI 刷新过于频繁，反而让观察能力下降

所以这一阶段必须先固定协议，再接 UI。

---

## 阶段 3：补齐可停止能力

### 目标

先让扩展建库“能安全停”，不要一上来就追求复杂续建。

### 范围

- `pause`
- `cancel`
- UI 停止入口
- 扩展 worker 接入 stop control

### 推荐实现方式

#### 3.1 先复用主线 worker control 协议

主线已经有：

- `pause_event`
- `cancel_event`
- `BuildCancelledError`

这套机制适合先复制到扩展控制面。

#### 3.2 只在安全边界检查控制信号

扩展侧不要为了“响应更快”把取消检查塞得到处都是。

应该只在这些边界检查：

- 来源扫描循环之间
- 文件解析循环之间
- 向量批写之间
- 阶段切换点

这样更稳，避免中途把某个文件或某批写入切成半状态。

#### 3.3 “真正可取消”依赖超时与回收

扩展侧的 `cancel` 不能只是 UI 事件。

对 `PDF / Tika` 而言，真正难停的地方通常是：

- 单个大文件解析
- `Tika` HTTP 请求
- sidecar 不响应

因此必须补：

- 单文件解析超时
- `Tika` 请求超时
- sidecar recycle / restart 策略

否则会出现“按钮能点，但任务半天停不下来”的假取消。

#### 3.4 任务冲突矩阵先于实现

在给扩展接 `pause / cancel / stop control` 之前，必须先把任务冲突矩阵定死。

尤其要防止：

- build 过程中用户切换 workspace
- build 过程中用户切换 data_root 预览值
- build 与 delete-index 同时进行
- build 与 watch 交错接管

### 验收标准

- 用户可主动取消扩展建库
- 取消后 UI 状态正确
- 取消后不会把主线状态带乱
- 取消后不会误显示构建完成

### 绝对不要做的事

- 不要在这一阶段直接把“取消”实现成“强杀线程”
- 不要靠 UI 直接改状态文件来伪造取消

---

## 阶段 4：补齐中断感知与软恢复

### 目标

在不引入高风险复杂 resume 的前提下，让扩展具备“知道自己上次没做完”的能力。

### 为什么先做软恢复

用户真正需要的第一层能力通常不是“从第 1732 个文件继续”，而是：

- 上次中断了，别让我毫无感知
- 告诉我现在应该继续还是重来

这是低风险、高收益的能力。

### 范围

- 扩展级 build state 文件
- 扩展级 interrupted / resumable 标记
- 启动后恢复提示
- “从头重建”与“尝试恢复”的选择入口

### 推荐实现方式

#### 4.1 记录扩展 build snapshot

至少记录：

- `pipeline`
- `workspace id`
- `source roots`
- `enabled formats`
- `manifest signature`
- `phase`
- `current cursor`
- `started_at`
- `updated_at`
- `interrupted`
- `build_id`
- `state_version`
- `last_good_build_id`

#### 4.2 首版恢复优先做“重新开始但可感知”

首版软恢复允许这样做：

- 发现上次异常中断
- 告诉用户“上次未完成”
- 用户可选择：
  - 从头重建
  - 清理状态
  - 仅查看日志

这已经比当前完全无感强很多。

#### 4.3 build state 必须带版本与原子写

扩展 build state 一旦引入，必须同时满足：

- 带 `state_version`
- 带 schema / parser 关键版本信息
- 原子写入
- 读到未知版本时可安全降级为“清理状态 + 重建”

#### 4.4 `last-good` 与恢复提示必须联动

恢复提示不能只告诉用户“上次中断了”，还必须告诉用户：

- 当前是否还有可服务的旧索引
- 如果继续恢复失败，是否还能退回旧索引
- 若旧索引不可用，必须明确写明“当前扩展查询不可用”

### 验收标准

- 扩展异常退出后，下次启动能感知上次中断
- UI 不再把异常中断误装成成功完成
- 用户有清晰的下一步动作

---

## 阶段 5：保守版真恢复

### 目标

在阶段 4 稳定之后，再补“真正继续做”的能力。

### 核心原则

首版真恢复只做：

- **文件级恢复**

不做：

- chunk 级恢复
- vector 精细恢复

### 推荐实现方式

#### 5.1 文件级 cursor

只记录：

- 已完成文件集合或文件级 cursor
- 当前来源目录签名
- 当前格式集签名
- 文件级 fingerprint（至少 `path + size + mtime`，最好再加轻量内容签名）

#### 5.2 store 与 vector 的恢复策略分离

最稳妥的做法是：

- store 可尝试文件级继续
- vector 阶段在首版恢复中允许保守地重建扩展 vector index

原因：

- vector 的一致性比速度更重要
- 宁可慢一点，也不要留下“表面 ready、实际残缺”的隐患

同时必须定义 durable commit 边界：

- 单文件何时算“可恢复”
- 恢复时哪些 store 写入允许复用
- 哪些 vector 状态必须丢弃重建

#### 5.3 只有签名匹配时才允许恢复

以下条件任一变化，都必须回退成“重新构建”：

- 来源目录变化
- 启用格式变化
- manifest signature 不匹配
- parser / schema 版本变化
- workspace 变化
- data_root 变化

#### 5.4 恢复不能只依赖开始时的 manifest

扩展 build 是长任务。

因此必须定义：

- 建库期间若来源目录继续发生变化，如何把这些变化记为 `pending`
- build 完成后，如何把 build 期间的新变化补齐

否则系统可能恢复成功，但还是漏掉建库期间新增或修改的文件。

### 验收标准

- 中途打断后，可在明确边界下恢复
- 恢复完成后的扩展索引结果与完整重建一致
- 不出现“续建成功但内容缺失”的 silent corruption

---

## 阶段 6：watchdog 与诊断闭环

### 目标

让扩展也具备主线同级别的“我知道自己卡了”的能力。

### 范围

- 扩展 watchdog
- stall report
- 诊断日志
- 用户可见的卡死提示

### 推荐实现方式

1. 复用主线 watchdog 的总体设计
2. 扩展独立维护自己的 watchdog state
3. 区分：
   - 真实无前进
   - 慢文件
   - sidecar 未响应
   - vector 写入停滞
4. watchdog 报告必须带 `build_id`、`workspace id` 与 `pipeline`

### 价值

这一步不是锦上添花，而是扩展路线非常必要的诊断基建，尤其是：

- `Tika` sidecar
- 大文件解析
- 长时间 vector 写入

---

## 是否需要“原子切换式建库”

### 当前判断

**不作为当前优先项。**

### 原因

1. 扩展当前首先缺的是“能稳定建、能看见、能停、能恢复”
2. 扩展本来就是独立命名空间，暂时没有主线那种“一旦切错影响全局”的压力
3. 现在就引入 staging / swap / dual version，会显著增加复杂度与测试面

### 后续何时再考虑

只有在以下情况同时出现时，才考虑：

- 扩展数据量已明显增大
- 用户强烈需要“旧扩展索引始终可用”
- 当前恢复与重建链已经稳定

---

## 最大风险与规避策略

### 风险 1：扩展状态机被改乱

#### 风险描述

一旦引入：

- paused
- cancelled
- interrupted
- resumable

这些新状态，就很容易出现：

- UI 状态
- 状态文件
- 实际索引状态

三者不一致。

#### 规避方式

- 先定义状态机，再写代码
- 让“真实数据状态”优先于“UI 展示状态”
- 每新增一个状态都要定义：
  - 进入条件
  - 退出条件
  - UI 呈现
  - 可执行动作

### 风险 2：resume 写坏索引一致性

#### 风险描述

这是本次改造最大的技术风险。

一旦：

- store cursor
- vector cursor
- UI 恢复状态

三边没对齐，就最容易出现“看起来成功，实际结果残缺”的隐性故障。

#### 规避方式

- 先软恢复，后真恢复
- 首版真恢复只做文件级
- vector 恢复先保守重建
- 恢复条件严格依赖签名匹配

### 风险 3：源码能跑，EXE 再次失真

#### 风险描述

这次 `PDF` 已经踩过一次。

#### 规避方式

- 每一阶段都把打包态验证并行纳入
- parser / dependency / frozen worker 改动必须同时做源码态与 EXE 态验证

### 风险 4：为抽象而抽象，连主线一起带崩

#### 风险描述

最危险的做法是：

- 一上来抽一个巨大通用 `BuildOrchestrator`
- 然后同时让主线与扩展一起迁移

#### 规避方式

- 先复制成熟模式到扩展
- 扩展稳定后，再考虑抽公共层

---

## 发布门禁与用户自救

### 发布门禁

以下任一条件不满足，都不应发布：

- 源码态通过、但 EXE 态未通过
- `PDF` 或 `Tika` 仍存在静默跳过且无明确错误
- 强杀 / 故障注入场景未覆盖
- `READY` 判定仍可能与真实数据状态不一致
- `cancel / interrupted / resumable` 状态仍可能互相混淆

### 阶段交付门禁

不仅正式发布要设门禁，每一个里程碑也必须设门禁。

后续实现时，任何里程碑只有在同时满足以下条件时，才允许进入下一阶段：

- 本阶段目标功能已经闭环，不是“先能跑起来”
- 本阶段涉及的 UI、状态文案、错误提示、用户动作入口已经闭环
- Markdown 主线没有回归
- 扩展已知高严重度问题为零
- 扩展已知中严重度问题如果仍存在，必须要么在本阶段修完，要么明确降级并阻止进入下一阶段
- 源码态与 EXE 态都已通过本阶段专项回归
- 本阶段新增状态、合同、妥协已经写回计划文档或 `ARCHITECTURE.md`

### 不允许的实施方式

后续执行时，绝对不允许以下做法：

- “功能先能跑，细节以后补”
- “状态先占位，后面再收口”
- “UI 先糊一个，文案以后再改”
- “先并到主线，测试以后补”
- “先把恢复链接上，已知错乱等下一阶段一起修”
- “先交给用户试，靠用户帮忙补基础缺陷”

### 已知缺陷处理原则

如果某一阶段结束时仍发现缺陷，必须按严重度处理：

- `P0 / P1`：不得进入下一阶段，不得视为阶段完成
- `P2`：默认应在本阶段收口；若确需延期，必须有明确 guardrail、明确用户影响边界、明确后续修复项
- `P3`：可延期，但必须记录且不得影响当前阶段的正确性承诺

这里的核心原则是：

> **每个里程碑完成时，都必须已经是一个“可交付的小成品”，而不是一个等着下阶段补锅的半成品。**

### 用户自救路径

扩展建库强化完成后，用户至少应有以下自救能力：

- 一键打开扩展日志目录
- 一键打开 watchdog / diagnostics 目录
- 一键清理扩展 build state / resume state
- 一键清理扩展索引并从头重建

这能显著降低后续支持成本，也能避免用户只能手工删除目录。

## 后续执行时的推荐工作法

后续真正开始改这条线时，推荐严格遵守下面的执行方式。

### 1. 每次只推进一个里程碑

不要把：

- parser 修复
- progress 大改
- cancel / resume
- watchdog

塞进一次提交。

### 2. 每个里程碑都要单独做打包验证

不能只看源码态。

### 2.5 每个里程碑都必须达到“可交付质量”

这里的“完成”不是指代码分支里已经能演示，而是指：

- 用户从界面进入该阶段涉及的功能时，不会遇到明显逻辑断口
- 状态、日志、文案、按钮行为已经和真实数据状态一致
- 已知严重问题已经清零
- 已知会误导用户的问题已经清零

如果某个里程碑结束时仍需要说：

- “这个先别用”
- “这里先忽略一下”
- “这个状态其实还不准”
- “这个打包版还没验”

那就不应视为该里程碑已完成。

### 3. 先把扩展内部跑顺，再考虑抽象主线 / 扩展公共层

复用应该先用“复制成熟模式”的方式落地，再在后续版本考虑抽公共基建。

### 4. 所有状态新增都要先写状态机表

后续执行时，建议每新增一个状态都在本文件或 `ARCHITECTURE.md` 里补一张状态机说明。

### 5. 发现跨阶段缺陷时，立即止损回收

如果某一阶段实施中暴露出会影响前一阶段正确性的缺陷：

- 不得带着它继续推进下一阶段
- 必须先回收并修正当前阶段
- 必须先补测试与文档，再继续后续工作

否则很容易把一条本可控的扩展强化线，做成连续几阶段都在补前面遗留错误的链式返工。

## 最终验收定义

只有当以下条件同时成立时，才算“扩展建库强化”真正完成。

### 功能面

- `PDF` 在源码态与 EXE 态都能稳定建库
- `Tika` 在源码态与 EXE 态都能稳定建库
- 全局扩展建库与单目录扩展建库都有可理解的进度
- 扩展建库至少支持取消与异常中断感知
- 用户可清楚区分：
  - 正常跳过
  - 真失败
  - 仍在进行
  - 已中断
  - 可恢复

### 架构面

- 主线 Markdown 不被回归
- 扩展隔离边界仍成立
- 状态机清晰
- 打包链闭环

### 体验面

- 用户不再需要靠猜测判断扩展是否卡死
- 用户不再需要靠“删目录重来”来处理大部分扩展建库中断
- 用户不会再看到“明明没建好却显示 ready”这类不诚实状态

## 一句话总纲

这次“扩展建库强化”的正确做法不是大重构，也不是零碎修补，而是：

> **先修扩展解析链与打包依赖，随后把主线已经验证成熟的建库控制面逐步下沉到扩展侧，再以保守方式补上可见、可停、可感知中断、可恢复的能力。**

在这个总纲下：

- `PDF / Tika` 继续保持自己的数据面与命名空间；
- 主线 Markdown 保持稳定，不被牵连；
- 扩展建库逐步从“实验性可跑”提升到“接近主线成熟度”的工业级子系统。
