# 检索优化计划

## 目标

本计划的目标不是继续堆叠检索复杂度，而是在保持产品轻量、稳定、通用的前提下，持续降低用户手工筛片段的时间成本，并提高前几条结果的真实可用度。

本轮执行的硬约束：

- 未经明确要求，不构建 EXE。
- UI 只负责展示、配置和状态反馈，不承担检索策略本身。
- 所有新增增强能力默认可关闭，不能绑死主流程。
- `reranker` 固定选型为 `BAAI/bge-reranker-v2-m3`。
- `ONNX INT8` 当前不纳入实施范围。

## 第一阶段：结果层去冗余与“新信息增益”选择

### 目标

- 压掉同页重复、父子覆盖、近重复片段。
- 保留定义、指标、例子、结论这类互补信息。
- 降低用户在同页大量片段里人工挑选的负担。

### 已完成

- 新增 `omniclip_rag/query_runtime.py`，统一承接：
  - 同页去冗余
  - 新信息增益选择
  - 查询条数推荐
- `service.py` 不再散落维护旧的同页去重逻辑；统一走 `query_runtime.select_query_hits(...)`。
- `gui.py` 只消费结构化 `QueryResult / QueryInsights`，不再自己维护一套片段选择策略。
- 同页片段选择改为“压重复、保互补”，而不是粗暴按同页减分或平均分。
- 已补回归测试：
  - `tests/test_query_runtime.py`
  - `tests/test_service.py`
  - `tests/test_gui.py`

### 本阶段刻意不做

- 不做激进去重，不把“同页只保留一个”写死。
- 不做个性化用户偏好学习，避免策略先天偏向某一类使用习惯。

### 是否需要重建

- 不需要。

## 第二阶段：混合召回融合升级

### 目标

- 让标题命中、路径命中、正文命中、FTS/BM25、向量召回、父级语义继承更稳定地协同。
- 避免某一个信号把其他信号彻底压死。
- 强化短词、概念词、多条件查询的差异化处理。

### 已完成

- 新增 `omniclip_rag/retrieval_policy.py`，把混合召回的查询分型、融合打分、候选池扩张从 `service.py` 抽离出来。
- 新增查询类型分型：
  - 单字
  - 短关键词
  - 概念词
  - 自然语言问题
- 按查询类型动态调整：
  - 是否启用向量召回
  - 词法/语义权重
  - 候选池规模
  - hydration 步长
- 引入稳定的融合补偿，避免单一检索源先天赢麻。
- `service.query()` 现在先构建 `QueryProfile`，再调用 `rank_candidates(...)`，使查询策略与 UI 彻底解耦。
- 已补专项测试：
  - `tests/test_retrieval_policy.py`
  - `tests/test_service.py`

### 本阶段刻意不做

- 不引入重型外部检索栈，不把本地桌面工具复杂化成服务端系统。
- 不做大规模手工同义词库，避免策略僵化和维护失控。

### 是否需要重建

- 不需要。

## 第三阶段：可选 reranker 安全接入

### 目标

- 在不破坏现有主流程的前提下，为概念型查询提供更强的精排能力。
- 让 `reranker` 成为可选增强，而不是基础依赖。

### 固定方案

- 模型：`BAAI/bge-reranker-v2-m3`
- 默认：关闭
- 下载：必须由用户显式触发
- 位置：`配置 -> 设置`
- 设备：
  - `cpu` 只用 CPU
  - `gpu/cuda` 只用 GPU
  - `auto` 自动判断

### 已完成

- 新增 `omniclip_rag/reranker.py`，定义可替换的 reranker 边界，避免把重排逻辑写死进 GUI 或主查询流程。
- 新增 `NullReranker` 与 `CrossEncoderReranker`，默认关闭时不引入额外开销。
- 新增本地模型就绪检查和显式下载入口；未下载时不会误触发网络访问。
- 查询链路已支持可选 rerank：
  - batching
  - 长文本截断
  - GPU OOM 熔断
  - 自动降批
  - 必要时自动回退 CPU
  - 再失败则直接跳过 reranker，不会让查询失败
- GUI 已接入：
  - reranker 开关
  - reranker 模型名
  - CPU/GPU batch size
  - 手动下载按钮
- 已补回归测试：
  - `tests/test_reranker.py`
  - `tests/test_service.py`
  - `tests/test_gui.py`

### 本阶段刻意不做

- 不把 reranker 设为默认开启。
- 不引入 ONNX INT8 双路线，避免这轮把运行时和维护复杂度一起抬高。
- 不把 reranker 变成建库前置条件；它只影响查询时精排。

### 是否需要重建

- 通常不需要。

## 第四阶段：候选条数“自适应推荐范围”

### 目标

- 降低用户反复试错“查询条数多少合适”的成本。
- 给出设备与运行历史相关的推荐区间，而不是偷偷改配置。

### 已完成

- 已持久化 `query_runtime.json` 运行历史。
- 已能根据设备模式、耗时、候选水位、是否启用 reranker 给出推荐区间。
- 已把 reranker 场景纳入推荐基线：
  - CPU 默认推荐 15 附近
  - GPU 默认推荐 50 附近
- 已把 reranker 降级/OOM 纳入推荐依据，避免用户把候选池开到明显不合理的范围。
- GUI 已接入：
  - 推荐区间展示
  - tooltip 原因说明
  - 推荐值一键应用
- `查询条目数` 与 reranker 的候选池语义已经联动，提示文案同步更新。

### 本阶段刻意不做

- 不自动偷偷改写用户配置；只给推荐，不强制覆盖。
- 不把推荐逻辑做成黑盒“神谕”，保留原因码解释。

### 是否需要重建

- 不需要。

## 第五阶段：AI 协作检索模式

### 目标

- 让用户尽量只做“提问 / 检索 / 粘贴回 AI”这几步。
- 让 AI 自己忽略低相关片段，并在上下文不足时返回下一轮关键词。

### 已完成

- 新增可选 `ai-collab` 导出模式，默认导出仍保持纯净。
- 协作模式只附加极简说明，不把产品做成内置聊天壳。
- 协作导出文案已做中英文兼容；中文查询场景会自动保持中文说明，不被系统英文环境带偏。
- GUI 已接入协作导出开关。
- 已补回归测试：
  - `tests/test_service.py`
  - `tests/test_gui.py`

### 本阶段刻意不做

- 不硬塞长尾部 prompt，不替用户绑定某一家 AI。
- 不在主程序里内置聊天窗口，继续保持和外部 AI 工具解耦。

### 是否需要重建

- 不需要。

---

## 必须保留的优点

- 块级输出
- 原格式高还原
- 默认轻量
- 不强绑某家 AI
- 不把主程序做成重聊天壳
- 不让 reranker 成为基础依赖
- UI 与功能代码保持尽量独立解耦

## 必须压掉的弊端

- 同页大量重复片段
- 用户自己判断哪些片段值得选
- 大候选池时 CPU/GPU 试错成本高
- reranker 带来的 OOM / 闪退风险
- 未来模型更新把主程序绑死

## 本轮交付后的实际结果

- 查询后的“人肉筛片段”成本已经明显从 UI 手工操作转移到后端选择和可选 AI 协作模式。
- 检索策略、运行时建议、reranker、AI 协作导出都已从界面逻辑中拆出，后续继续迭代时更容易保证稳定性。
- 所有阶段都以“不需要重建索引”为前提落地；只有未来如果切块规则或向量模型本身发生变化，才需要重建。

## 后续增量方向（不属于本轮交付）

- 继续微调混合召回和 reranker 权重。
- 观察真实大库、大页场景下的 CPU/GPU 推荐区间是否需要继续校准。
- 如未来 CPU reranker 成为高频诉求，再单独评估 ONNX INT8 路线。
